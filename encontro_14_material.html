1 Contar registros com COUNT

Às vezes você só precisa saber quantos registros existem, sem precisar trazer todos eles. O SQL tem uma função para isso: COUNT(*), que retorna o número de linhas que combinam com a consulta.

Como fazer:

  1. Crie uma rota nova, por exemplo GET /seuRecurso/total
  2. Use SELECT COUNT(*) as total FROM seuRecurso com .get() — já que o resultado é uma única linha
  3. Atenção: essa rota precisa vir antes de GET /:id no arquivo, senão "total" seria interpretado como um ID (mesmo problema de ordem de rotas que vocês já viram no E04)
router.get('/total', (req, res, next) => {
  try {
    const resultado = db.prepare('SELECT COUNT(*) as total FROM produtos').get()
    res.json(resultado) // { total: 5 }
  } catch (err) { next(err) }
})
2 Garantir unicidade com UNIQUE

Algumas colunas não devem se repetir — um e-mail, um código de produto. O SQLite tem a restrição UNIQUE, que impede dois registros com o mesmo valor naquela coluna.

Como fazer (se fizer sentido para o seu recurso):

  1. No database.js, adicione UNIQUE depois do tipo da coluna que deve ser única
  2. Apague o banco.db e reinicie o servidor para recriar a tabela com a nova restrição (mesma situação do E11)
  3. Teste tentando criar dois registros com o mesmo valor naquela coluna — o segundo deve falhar
// Exemplo: impedir dois produtos com o mesmo código
db.exec(`
  CREATE TABLE IF NOT EXISTS produtos (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    codigo TEXT UNIQUE NOT NULL,
    nome TEXT NOT NULL,
    preco REAL NOT NULL
  )
`)

Ao testar uma inserção duplicada, o banco lança um erro — que o seu try/catch já está preparado para capturar e devolver via middleware de erros (E08), mostrando que tudo que você construiu até aqui se conecta.

Índices — acelerando buscas em tabelas grandes
Quando uma tabela tem milhares ou milhões de linhas, buscar por uma coluna específica pode ficar lento — o banco precisa "olhar" linha por linha. Um índice (criado com CREATE INDEX) é uma estrutura auxiliar que acelera buscas em colunas específicas, parecida com o índice remissivo no final de um livro. A coluna id com PRIMARY KEY já ganha um índice automaticamente — mas colunas usadas com frequência em WHERE (como um e-mail de login) se beneficiam de índices próprios.
→ SQLite — CREATE INDEX
Ferramentas de administração de banco em produção
O DB Browser que vocês usam é ótimo para desenvolvimento local. Em produção, equipes costumam usar ferramentas como TablePlus, DBeaver ou os painéis administrativos integrados de plataformas como Railway e Supabase — que permitem inspecionar, consultar e até editar dados de bancos rodando remotamente, com mais recursos de segurança e colaboração entre equipe.
