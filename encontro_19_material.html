3 Sprint — finalizar o que falta

2
Trabalhar nos itens pendentes do checklist
Priorize o que bloqueia a demo — pequenos ajustes depois

Se você identificou itens faltando no checklist, resolva nessa ordem de prioridade:

1. Prioridade alta — servidor não inicia, rota retornando erro 500, banco com estrutura errada

2. Prioridade média — validação faltando, 404 retornando 200, JOIN sem dados

3. Prioridade baixa — mensagens de erro com texto genérico, commits com mensagens vazias

Não esconda erros para a demo Se uma rota não está funcionando, seja transparente durante a apresentação. Mostrar que você entende o que está errado vale mais do que esconder o problema.

4 Demo — como apresentar em 3 minutos

A demo é ao vivo no Postman — você mostra o sistema funcionando, não explica o código. Siga este roteiro:

min 1
Apresentar o projeto e criar dados
Diga em uma frase o que o sistema faz. Depois execute no Postman: POST para criar 1 registro na tabela principal → mostrar o 201 e o id gerado.
POST /pai/:id/filhos para criar um filho vinculado → mostrar o campo FK preenchido.
min 2
Mostrar leitura e o JOIN
GET /recurso → listar todos.
GET /pai/:id → mostrar o objeto pai com o array de filhos embutido (JOIN).
GET /recurso/999 → mostrar o 404 com mensagem descritiva.
min 3
Mostrar atualização, remoção e validação
PUT /recurso/1 → atualizar um campo → mostrar o dado atualizado na resposta.
POST sem campos obrigatórios → mostrar o 400 com a mensagem de erro.
DELETE /recurso/1 → 204. GET /recurso/1 logo depois → 404 confirmando remoção.
Antes de chamar o docente para a demo Faça o roteiro completo sozinho uma vez para garantir que todos os requests estão funcionando. Resete o banco se necessário (apague banco.db e reinicie o servidor) para que a demo comece com dados limpos.
⭐ Quer ir além? — para quem terminou antes do tempo
1 Escrever o README.md do projeto

Um README bem escrito é o cartão de visitas do repositório. Inclua: o que o sistema faz, como instalar e rodar localmente, quais são as rotas disponíveis com exemplos de request/response, e quais variáveis de ambiente são necessárias (referencie o .env.example).

2 Adicionar filtros às listagens

Estender o GET /recurso para aceitar query params como filtros:

GET /tarefas?concluida=0         → só tarefas pendentes
GET /tarefas?prioridade=alta    → só tarefas de alta prioridade
GET /anuncios?categoria_id=2    → anúncios de uma categoria

No Express, os query params chegam em req.query. Monte o WHERE dinamicamente com base nos params presentes.

3 Testes automatizados no Postman

O Postman permite escrever scripts de teste que verificam o status code e o corpo da resposta automaticamente. Adicione testes básicos nas requests já criadas:

// Na aba "Tests" de cada request no Postman
pm.test("Status 201", () => pm.response.to.have.status(201))
pm.test("Tem id", () => {
  const body = pm.response.json()
  pm.expect(body.id).to.be.a("number")
})

Depois use o Collection Runner do Postman para rodar todos os testes em sequência e ver o relatório.

OpenAPI / Swagger — documentação interativa da API
O OpenAPI é um padrão para descrever APIs REST em YAML ou JSON. O Swagger UI gera uma interface interativa onde qualquer pessoa pode ver e testar as rotas sem precisar do Postman. É o padrão de documentação de API mais usado em empresas. A biblioteca swagger-ui-express integra isso ao Express em poucas linhas.
→ swagger.io — OpenAPI Specification
Rate limiting — proteger a API contra abuso
Sem proteção, qualquer pessoa pode fazer milhares de requisições por segundo para a sua API. O express-rate-limit adiciona um limite de requisições por IP (ex: máx 100 por 15 min) em uma linha de middleware. É um dos primeiros recursos de segurança adicionados em APIs em produção.
→ express-rate-limit no npm
