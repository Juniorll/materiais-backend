1 Os comandos SQL de hoje

INSERT — adicionar uma linha à tabela O comando INSERT INTO adiciona um novo registro (uma linha) a uma tabela. Você especifica quais colunas vai preencher e quais valores entram em cada uma.
INSERT INTO nome_da_tabela (coluna1, coluna2) VALUES (valor1, valor2)
SELECT — buscar linhas de uma tabela O comando SELECT busca dados de uma tabela. O asterisco * significa "todas as colunas". A cláusula WHERE é opcional e filtra quais linhas devem ser retornadas.
SELECT * FROM nome_da_tabela SELECT * FROM nome_da_tabela WHERE coluna = valor
Prepared statements — por que usar ? em vez de colar o valor direto Em vez de montar a string SQL concatenando valores diretamente, usamos um placeholder (o símbolo ?) no comando, e passamos os valores reais separadamente. Isso evita um problema sério de segurança chamado SQL Injection — quando alguém manda dados maliciosos disfarçados de texto comum para manipular o comando SQL. O better-sqlite3 cuida de tudo isso automaticamente, desde que você use ? em vez de concatenar strings.
Nunca concatene valores direto no SQL Código como `INSERT INTO produtos VALUES ('${nome}')` parece funcionar, mas é uma porta aberta para ataques. Sempre use ? como placeholder — você vai ver exatamente como fazer isso nos próximos passos.

2 Manual — INSERT e SELECT no código

1
Entender db.prepare() e os métodos .run() e .all()
A forma como o better-sqlite3 executa comandos SQL

O better-sqlite3 tem um padrão de uso: primeiro você prepara o comando SQL com db.prepare(), depois executa com um dos métodos abaixo, dependendo do que você precisa:

MétodoQuando usarO que retorna
.run()INSERT, UPDATE, DELETEInformações sobre a execução (ex: o id gerado)
.all()SELECT que retorna várias linhasUm array de objetos
.get()SELECT que retorna uma linha sóUm único objeto, ou undefined
Por que .all() e .get() são diferentes Quando você busca uma lista (ex: todos os produtos), quer um array — use .all(). Quando busca um item específico por ID, quer só um objeto — use .get(), que já entrega o resultado sem precisar pegar a posição [0] do array.
2
Atualizar a rota GET para buscar do banco
Substituindo o array em memória por uma consulta SQL real

Abra o arquivo de rotas do seu recurso, em routes/. No topo do arquivo, importe a conexão com o banco:

const express = require('express')
const router = express.Router()
const db = require('../database') // ../ porque routes/ está uma pasta abaixo da raiz

Agora substitua a rota GET principal. Em vez de res.json(array), faça a consulta SQL:

// Exemplo com a tabela "produtos" — adapte para o nome do seu recurso
router.get('/', (req, res, next) => {
  try {
    const produtos = db.prepare('SELECT * FROM produtos').all()
    res.json(produtos)
  } catch (err) {
    next(err)
  }
})
Repare que o try/catch que você aprendeu no E08 continua presente. Operações de banco de dados podem falhar — arquivo corrompido, tabela inexistente — então capturar erros aqui é tão importante quanto era nas rotas com array.

4
Atualizar a rota POST para inserir no banco
Com prepared statements — usando ? como placeholder

Substitua a rota POST. Em vez de criar o objeto e dar push() no array, você executa um INSERT:

router.post('/', (req, res, next) => {
  try {
    const { nome, preco } = req.body

    // Os ? são substituídos, na ordem, pelos valores do array em .run()
    const resultado = db.prepare('INSERT INTO produtos (nome, preco) VALUES (?, ?)')
      .run(nome, preco)

    // resultado.lastInsertRowid é o id que o AUTOINCREMENT acabou de gerar
    const novoProduto = db.prepare('SELECT * FROM produtos WHERE id = ?')
      .get(resultado.lastInsertRowid)

    res.status(201).json(novoProduto)
  } catch (err) {
    next(err)
  }
})
Por que buscar o produto de novo depois do INSERT? O .run() não devolve o registro inteiro — só metadados da operação, como o lastInsertRowid (o ID que acabou de ser gerado pelo AUTOINCREMENT). Para devolver o objeto completo ao cliente — como você fazia no array — fazemos um SELECT rápido logo em seguida, buscando pelo ID recém-criado.
Nunca mais calcular ID manualmente Lembra do bug de IDs duplicados depois de deletar itens, que vimos com o array? Isso acabou. O AUTOINCREMENT do SQLite garante IDs únicos sempre, mesmo depois de exclusões — sem nenhum código adicional.

1 Buscar um item específico por ID, agora com SQL

Você já tem a rota GET /:id usando array.find() dos encontros anteriores. Migre ela para usar o banco também, usando .get() em vez de .all() — já que você está buscando um único registro.

Como fazer:

  1. Use WHERE id = ? no SQL, passando o ID convertido com Number() como antes
  2. Use .get() em vez de .all() — ele retorna o objeto direto, ou undefined se não encontrar
  3. Mantenha a verificação if (!item) return res.status(404)... — ela continua necessária
router.get('/:id', (req, res, next) => {
  try {
    const id = Number(req.params.id)
    const produto = db.prepare('SELECT * FROM produtos WHERE id = ?').get(id)
    if (!produto) return res.status(404).json({ erro: 'Não encontrado' })
    res.json(produto)
  } catch (err) { next(err) }
})
2 Filtro por query param, agora com SQL

Você já tinha um filtro por query param usando array.filter(). Adapte para usar WHERE no SQL — o resultado para o cliente é idêntico, só muda de onde vem.

router.get('/', (req, res, next) => {
  try {
    const { categoria } = req.query

    if (categoria) {
      const filtrados = db.prepare('SELECT * FROM produtos WHERE categoria = ?').all(categoria)
      return res.json(filtrados)
    }

    const produtos = db.prepare('SELECT * FROM produtos').all()
    res.json(produtos)
  } catch (err) { next(err) }
})

SQL Injection — o ataque que os placeholders previnem
SQL Injection é uma das vulnerabilidades mais antigas e ainda mais comuns em aplicações web. Acontece quando um sistema concatena entrada do usuário diretamente num comando SQL, permitindo que um atacante "injete" comandos próprios — por exemplo, enviando como nome de usuário algo como ' OR '1'='1, que pode burlar uma verificação de login inteira. Os prepared statements que você usou hoje (com ?) eliminam esse risco automaticamente, porque tratam os valores como dados, nunca como parte do comando SQL.
→ OWASP — SQL Injection
Transactions — agrupando várias operações em uma só
Imagine um sistema de transferência bancária: tirar dinheiro de uma conta e colocar em outra precisam acontecer juntos — se uma falhar, a outra também deve ser desfeita. Isso se chama transaction (transação): um grupo de operações SQL que são tratadas como uma unidade só, tudo ou nada. O better-sqlite3 suporta isso com db.transaction(). É um conceito mais avançado, mas fundamental em sistemas que envolvem dinheiro ou múltiplas tabelas dependentes.
→ Documentação do better-sqlite3 — Transactions
