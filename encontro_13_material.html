1 Os comandos de hoje — e o cuidado que eles exigem

UPDATE — alterar registros existentes O comando UPDATE modifica os valores de uma ou mais linhas já existentes na tabela. Você especifica quais colunas mudam (SET) e quais linhas são afetadas (WHERE).
UPDATE nome_da_tabela SET coluna1 = novo_valor1, coluna2 = novo_valor2 WHERE coluna = valor
DELETE — remover registros O comando DELETE remove linhas inteiras da tabela. Diferente do UPDATE, ele não precisa especificar colunas — só quais linhas remover.
DELETE FROM nome_da_tabela WHERE coluna = valor
⚠ O perigo real: UPDATE e DELETE sem WHERE
Repare que nas duas sintaxes acima, o WHERE aparece como se fosse opcional na gramática do SQL — e tecnicamente é. Mas se você escrever DELETE FROM produtos sem nenhum WHERE, o banco entende isso como "delete TODAS as linhas da tabela produtos" — e executa, sem perguntar, sem confirmação. O mesmo vale para UPDATE: sem WHERE, TODAS as linhas recebem o novo valor.

Esse é um dos erros mais comuns — e mais destrutivos — que desenvolvedores cometem, até os experientes. Hoje você vai ver esse comportamento acontecendo de propósito, num ambiente seguro, para nunca esquecer.

2
Criar a rota PUT com SQL
UPDATE sempre acompanhado de WHERE

No arquivo de rotas, substitua a rota PUT (que ainda usa array.findIndex()) pela versão com SQL:

router.put('/:id', (req, res, next) => {
  try {
    const id = Number(req.params.id)
    const { nome, preco } = req.body

    // Verificar se o registro existe antes de atualizar
    const existe = db.prepare('SELECT * FROM produtos WHERE id = ?').get(id)
    if (!existe) return res.status(404).json({ erro: 'Não encontrado' })

    // UPDATE sempre com WHERE — nunca esqueça desta linha
    db.prepare('UPDATE produtos SET nome = ?, preco = ? WHERE id = ?')
      .run(nome, preco, id)

    // Buscar e devolver o registro já atualizado
    const produtoAtualizado = db.prepare('SELECT * FROM produtos WHERE id = ?').get(id)
    res.json(produtoAtualizado)
  } catch (err) {
    next(err)
  }
})
Por que verificar a existência antes de fazer o UPDATE? Diferente do array, onde o findIndex() já te dizia se o item existia, o UPDATE ... WHERE id = ? simplesmente não faz nada se nenhuma linha bater com o WHERE — sem gerar erro. Por isso fazemos um SELECT primeiro: assim conseguimos retornar 404 corretamente quando o ID não existe, em vez de responder 200 com um "sucesso" que não atualizou nada de fato.

4
Criar a rota DELETE com SQL
A mesma disciplina: sempre verificar, sempre usar WHERE
router.delete('/:id', (req, res, next) => {
  try {
    const id = Number(req.params.id)

    const existe = db.prepare('SELECT * FROM produtos WHERE id = ?').get(id)
    if (!existe) return res.status(404).json({ erro: 'Não encontrado' })

    // DELETE sempre com WHERE — nunca esqueça desta linha
    db.prepare('DELETE FROM produtos WHERE id = ?').run(id)

    res.status(204).send()
  } catch (err) {
    next(err)
  }
})
O padrão se repete Note que PUT e DELETE seguem exatamente a mesma estrutura: verificar existência → executar a operação com WHERE → responder adequadamente. Esse padrão consistente facilita memorizar e revisar código depois.
5
Testar o DELETE e fechar o ciclo de testes

DELETE em /seuRecurso/1 no Postman.

✓ Resultado esperado — status 204
Sem body na resposta. Faça um GET na lista completa — o item não aparece mais. Reinicie o servidor e confirme: ele continua ausente (porque foi removido do banco de verdade, não apenas da memória).

Checklist final do encontro: teste as quatro operações em sequência no Postman — POST (cria) → GET (lista, item aparece) → PUT (atualiza) → GET por ID (confirma mudança) → DELETE → GET (lista, item sumiu). Esse é o seu CRUD completo, 100% no banco de dados.

1 Busca parcial com LIKE

O filtro por query param que você já tem busca por igualdade exata (WHERE genero = ?). Para uma busca de texto parcial — como uma caixa de busca que encontra "Harry" dentro de "Harry Potter" — o SQL tem o operador LIKE, combinado com o símbolo % (que significa "qualquer coisa aqui").

Como fazer:

  1. Adicione um novo query param, por exemplo busca
  2. Monte o valor do placeholder com os símbolos de porcentagem em volta: `%${busca}%`
  3. Use LIKE ? no lugar de = ? no WHERE
if (busca) {
  const resultados = db.prepare('SELECT * FROM produtos WHERE nome LIKE ?')
    .all(`%${busca}%`)
  return res.json(resultados)
}

Teste com ?busca=ana — deve encontrar qualquer produto que tenha "ana" em qualquer parte do nome, em qualquer posição.

2 Ordenar resultados com ORDER BY

Por padrão, o SELECT retorna os registros numa ordem que não é garantida. O ORDER BY permite ordenar por qualquer coluna, em ordem crescente (ASC, padrão) ou decrescente (DESC).

Como fazer:

// Do mais barato para o mais caro
db.prepare('SELECT * FROM produtos ORDER BY preco ASC').all()

// Do mais caro para o mais barato
db.prepare('SELECT * FROM produtos ORDER BY preco DESC').all()

Experimente adicionar um query param ordenar que aceite asc ou desc, e monte a query SQL dinamicamente com base nesse valor.

Cuidado ao montar SQL dinâmico com ORDER BY Diferente de valores normais, o ASC/DESC não pode ser passado como ? (placeholder) — o SQL não aceita parâmetros nessa posição. Nesse caso específico, valide o valor manualmente antes de inserir na string (ex: só aceite exatamente as strings "asc" ou "desc", rejeitando qualquer outra coisa) para não abrir brecha de SQL Injection.

Soft delete — deletar sem realmente apagar
Em muitos sistemas profissionais, o DELETE nunca remove a linha de verdade — em vez disso, adiciona-se uma coluna como deletado_em, e o "delete" só preenche essa data. As consultas (SELECT) passam a ignorar linhas com essa coluna preenchida. Essa técnica se chama soft delete, e existe porque dados raramente devem ser perdidos para sempre — é comum precisar recuperar algo "deletado" por engano, ou manter histórico para auditoria.
→ Pesquise "soft delete pattern" para ver implementações em diferentes linguagens
Migrations — versionando mudanças na estrutura do banco
Quando você precisou adicionar uma coluna nova no E11 (no desafio do "Quer ir além"), teve que apagar o banco inteiro e recriar. Em produção, isso é inaceitável — apagaria todos os dados reais dos usuários. Ferramentas de migration (como as que vêm com Prisma ou Knex) resolvem isso: cada alteração na estrutura do banco vira um arquivo versionado e reversível, aplicado sem perder os dados existentes.
