1 Por que nunca salvar senha em texto puro

Nunca faça isso INSERT INTO usuarios (nome, email, senha) VALUES (?, ?, ?) com a senha direto do body. Se o banco vazar (e bancos vazam), todas as senhas dos seus usuários ficam expostas — e muitos usam a mesma senha em outros serviços. Você seria responsável por isso.
Como o bcrypt resolve isso O bcrypt transforma a senha em um hash — uma string longa e aparentemente aleatória. O processo é irreversível: não existe função "descryptografar o hash". Para verificar se uma senha está correta, o bcrypt refaz o cálculo e compara os hashes. Mesma senha, mesmo hash? Correto. Diferente? Errado. E o bcrypt adiciona um "sal" aleatório — duas instâncias do mesmo hash para a mesma senha sempre ficam diferentes, o que torna ataques de dicionário ineficazes.

2 Async/await — o bcrypt é assíncrono

Por que o bcrypt é async O bcrypt faz um cálculo intenso de propósito (para dificultar ataques de força bruta). Esse cálculo demora alguns milissegundos — tempo suficiente para travar o servidor se fosse síncrono. Por isso, bcrypt.hash() é assíncrono: ele devolve uma Promise, e você precisa de await para esperar o resultado.
→
O padrão async/await nas rotas com bcrypt

Adicione async antes do callback da rota e await antes do bcrypt.hash(). O try/catch já presente captura erros assíncronos também.

// Antes — rota síncrona (como você já conhece)
router.post('/', (req, res, next) => { ... })

// Depois — rota assíncrona para usar await
router.post('/', async (req, res, next) => {
  try {
    const senha_hash = await bcrypt.hash(senha, 10) // espera o resultado
    // continua aqui só depois que o hash estiver pronto
  } catch (err) { next(err) }
})
O que o 10 significa É o custo do bcrypt — quantas rodadas de processamento ele faz. Custo 10 é o padrão recomendado: demora ~100ms em hardware moderno (rápido o suficiente para o usuário, lento o suficiente para atacantes). Aumentar para 12 dobra o tempo; diminuir para 8 acelera mas reduz a segurança.

3 Adicionar a tabela usuarios ao banco

1
Adicionar a tabela usuarios ao database.js
A tabela de usuários é a mesma para todos os projetos

Adicione este bloco no final do seu database.js, após as tabelas de negócio do projeto:

// database.js — adicionar ao final, após as tabelas existentes
db.exec(`CREATE TABLE IF NOT EXISTS usuarios (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  nome TEXT NOT NULL,
  email TEXT UNIQUE NOT NULL,
  senha_hash TEXT NOT NULL
)`)
UNIQUE no email — dupla proteção Além de verificar na rota (e retornar um 409 descritivo), o TEXT UNIQUE garante que o banco nunca aceite dois emails iguais mesmo que a verificação da rota falhe por algum motivo. Sempre é bom ter as duas camadas.

Reinicie o servidor. No DB Browser clique em Refresh e confirme que a tabela usuarios apareceu na lista.

4 Implementar POST /auth/cadastro

2
Criar routes/auth.js com a rota de cadastro
// routes/auth.js
const express = require('express')
const router = express.Router()
const db = require('../database')
const bcrypt = require('bcrypt')
const { validarObrigatorios, emailValido } = require('../helpers/validacao')

// POST /auth/cadastro
router.post('/cadastro', async (req, res, next) => {
  try {
    const { nome, email, senha } = req.body

    // Validação dos campos
    const erros = validarObrigatorios(req.body, ['nome', 'email', 'senha'])
    if (email && !emailValido(email)) erros.push('email com formato inválido')
    if (senha && senha.length < 6) erros.push('senha deve ter pelo menos 6 caracteres')
    if (erros.length > 0) return res.status(400).json({ erros })

    // Verificar se email já está cadastrado
    const existe = db.prepare('SELECT id FROM usuarios WHERE email = ?')
      .get(email.toLowerCase().trim())
    if (existe) return res.status(409).json({ erro: 'Email já cadastrado' })

    // Gerar o hash da senha (custo 10)
    const senha_hash = await bcrypt.hash(senha, 10)

    // Salvar o usuário — nunca retornar o senha_hash
    const resultado = db.prepare(
      'INSERT INTO usuarios (nome, email, senha_hash) VALUES (?, ?, ?)'
    ).run(nome.trim(), email.toLowerCase().trim(), senha_hash)

    res.status(201).json({
      id: resultado.lastInsertRowid,
      nome: nome.trim(),
      email: email.toLowerCase().trim()
    })
  } catch (err) { next(err) }
})

module.exports = router
emailValido — adicionar ao helpers/validacao.js se ainda não estiver lá Se você não implementou o Extra 1 do E18, adicione agora: 
function emailValido(email) {
  return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)
}

5 Verificar e testar

4
Testar no Postman e confirmar no DB Browser

Execute estes cenários em ordem:

1. Cadastro válido — POST /auth/cadastro com body { "nome": "Teste", "email": "teste@email.com", "senha": "minhasenha123" } → deve retornar 201 com id, nome e email (sem senha_hash)

2. DB Browser — Refresh na tabela usuarios. O campo senha_hash deve mostrar uma string longa que começa com $2b$10$ — nunca a senha original

3. Email duplicado — repita o mesmo POST → deve retornar 409 com "Email já cadastrado"

4. Campos faltando — POST /auth/cadastro sem o campo senha → deve retornar 400

5. Senha curta — POST /auth/cadastro com "senha": "123" → deve retornar 400 com "senha deve ter pelo menos 6 caracteres"

⭐ Quer ir além? (opcional)
1 Validar força da senha

Além do comprimento mínimo, adicione regras mais robustas de senha no helpers/validacao.js:

function validarSenha(senha) {
  const erros = []
  if (senha.length < 8) erros.push('senha deve ter pelo menos 8 caracteres')
  if (!/[A-Z]/.test(senha)) erros.push('senha deve ter pelo menos uma letra maiúscula')
  if (!/[0-9]/.test(senha)) erros.push('senha deve ter pelo menos um número')
  return erros
}

Use no cadastro: erros.push(...validarSenha(senha))

2 Rota GET /auth/usuarios — listar usuários sem expor o hash

Crie uma rota de listagem que retorna usuários sem o campo senha_hash. Use SELECT id, nome, email em vez de SELECT * — bom hábito que se aplica a qualquer campo sensível:

router.get('/usuarios', (req, res, next) => {
  try {
    const usuarios = db.prepare('SELECT id, nome, email FROM usuarios').all()
    res.json(usuarios)
  } catch (err) { next(err) }
})

Estes tópicos não são cobrados e não entram no projeto.
Argon2 — mais seguro que bcrypt
O bcrypt foi criado em 1999 e ainda é seguro para a maioria dos casos. Mas o OWASP (organização de referência em segurança web) recomenda o Argon2 como primeira escolha para hash de senhas desde 2019. O Argon2 ganhou o Password Hashing Competition, é mais resistente a ataques com hardware especializado (GPUs/ASICs) e tem parâmetros mais flexíveis. A biblioteca argon2 no npm tem API praticamente idêntica ao bcrypt.
→ OWASP Password Storage Cheat Sheet
User enumeration — revelar se um email existe
Retornar "Email já cadastrado" revela que aquele email está cadastrado no sistema — o que pode ser um problema de privacidade em sistemas sensíveis (saúde, finanças). A técnica usada nesses casos é responder sempre com a mesma mensagem genérica, independente do motivo do erro, e verificar o status de forma assíncrona por email. Para a maioria dos sistemas, retornar o erro específico é aceitável e mais usável.
Verificação de email — confirmar antes de ativar
Em vez de ativar a conta imediatamente, sistemas mais robustos enviam um email com um link de confirmação. O usuário só consegue fazer login depois de confirmar. O fluxo envolve gerar um token de verificação (UUID ou hash), salvar no banco com validade, enviar por email e criar uma rota GET /auth/verificar/:token que ativa a conta.
