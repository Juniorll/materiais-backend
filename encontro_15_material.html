1 PUT /:id — atualizar um registro no banco

Como o PUT funciona O cliente envia os novos dados no body. A rota busca o registro pelo id — se não existir, retorna 404. Se existir, executa o UPDATE no banco e retorna o registro já com os valores atualizados. Campos não enviados mantêm o valor original usando o operador ?? (nullish coalescing).
1
Adicionar a rota PUT /:id no arquivo de rotas do projeto
Localize o seu projeto e copie o bloco correspondente
🛒 Sistema de Pedidos / Loja — tabela: pedidos
// PUT /pedidos/:id
router.put('/:id', (req, res, next) => {
  try {
    const existente = db.prepare('SELECT * FROM pedidos WHERE id = ?').get(req.params.id)
    if (!existente) return res.status(404).json({ erro: 'Pedido não encontrado' })

    const { cliente, status, total } = req.body
    db.prepare('UPDATE pedidos SET cliente = ?, status = ?, total = ? WHERE id = ?')
      .run(cliente ?? existente.cliente, status ?? existente.status, total ?? existente.total, req.params.id)

    const atualizado = db.prepare('SELECT * FROM pedidos WHERE id = ?').get(req.params.id)
    res.json(atualizado)
  } catch (err) { next(err) }
})

O que o ?? faz O operador ?? retorna o valor da esquerda se ele não for null nem undefined. Ou seja: titulo ?? existente.titulo usa o novo título se ele vier no body — e mantém o valor atual se não vier. Isso permite que o cliente envie só os campos que quer alterar.

2 DELETE /:id — remover um registro do banco

Como o DELETE funciona — e o que é o status 204 A rota busca o registro pelo id — se não existir, retorna 404. Se existir, executa o DELETE no banco e responde com o status 204 No Content: indica sucesso, mas sem corpo na resposta. Tentar retornar um JSON depois do 204 vai causar erro — a resposta não tem corpo por definição.

⭐ Quer ir além? (opcional)
1 PATCH — atualização parcial explícita

O PUT substitui o recurso inteiro (mesmo que você use ?? para manter campos). O PATCH é semanticamente mais correto para atualizações parciais — ele atualiza só o que você manda. O código é praticamente igual ao PUT, só troca o método:

router.patch('/:id', (req, res, next) => { /* mesma lógica do PUT */ })

Na prática muitas APIs usam só PUT por simplicidade. O importante é ser consistente.

2 Soft delete — marcar como deletado sem remover

Em vez de apagar o registro do banco, adicione uma coluna deletado INTEGER DEFAULT 0 e no DELETE faça um UPDATE para deletado = 1. O GET então filtra WHERE deletado = 0. Isso preserva o histórico e permite "desfazer" uma exclusão — padrão usado em sistemas que precisam de auditoria.

Idempotência — por que PUT é diferente de POST
Chamar PUT /recurso/1 dez vezes com o mesmo body produz sempre o mesmo resultado — isso se chama idempotência. Chamar POST /recurso dez vezes cria dez registros — não é idempotente. Essa distinção define quando usar cada método e é parte do design de APIs RESTful.
Transações no banco — ou tudo, ou nada
Se uma operação precisa fazer múltiplos INSERTs ou UPDATEs de forma atômica (ou todos funcionam, ou nenhum é salvo), usamos transações. O better-sqlite3 suporta isso com db.transaction(). Essencial em operações financeiras, pedidos com itens, e qualquer fluxo onde metade do resultado é inaceitável.
