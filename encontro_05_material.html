1 Completando o CRUD — PUT e DELETE

Você já conhece GET e POST. Hoje entram as duas operações que faltam:

PUT — atualizar um recurso existente PUT recebe o ID do item pelo parâmetro de rota e os novos dados pelo body. O servidor encontra o item, substitui seus dados pelos novos e devolve o item atualizado. Se o item não existir, retorna 404. Ao contrário do POST, o PUT não cria — ele só atualiza algo que já existe.
DELETE — remover um recurso DELETE recebe o ID pelo parâmetro de rota e remove o item correspondente. Não precisa de body — o ID na URL é suficiente. Quando bem-sucedido, a convenção é retornar o status 204 No Content: a operação funcionou, mas não há nada para devolver (o item foi removido).
findIndex() — necessário para PUT e DELETE Nos encontros anteriores você usou array.find() para buscar um item — ele retorna o objeto ou undefined. Para atualizar e deletar, você precisa da posição do item no array, não o item em si. É aí que entra o array.findIndex(): ele retorna o índice (posição) do item no array, ou -1 se não encontrar. Com o índice em mãos, você consegue substituir (array[index] = novoItem) ou remover (array.splice(index, 1)).

2 Status codes — o idioma das respostas HTTP

Cada resposta do servidor carrega um código de status que indica o que aconteceu. Usar o código correto é parte do design de uma boa API. Estes são os que você vai usar neste encontro — e na maioria dos projetos reais:

200
OK — operação bem-sucedida com dados para retornar
201
Created — novo recurso criado com sucesso (POST)
204
No Content — operação bem-sucedida sem nada a retornar (DELETE)
404
Not Found — o recurso solicitado não existe

3 Manual — adicionando PUT e DELETE ao projeto

A partir deste encontro, você trabalha diretamente no index.js do seu projeto. Veja o exemplo com uma API de contatos e em seguida adapte para o seu recurso.

1
Criar a rota PUT /:recurso/:id
Atualizar um item existente pelo ID

O PUT combina o que você já sabe: recebe o id pelo parâmetro de rota (como no GET por ID) e recebe os novos dados pelo body (como no POST). A diferença é que em vez de criar algo novo, você substitui os dados de um item que já existe.

O findIndex() retorna a posição do item no array. Se retornar -1, o item não existe. Se encontrar, a linha contatos[index] = {...} substitui o objeto naquela posição.

Exemplo com contatos — adapte para o recurso do seu projeto:

app.put('/contatos/:id', (req, res) => {
  const id = Number(req.params.id)
  const index = contatos.findIndex(c => c.id === id)

  if (index === -1) {
    return res.status(404).json({ erro: 'Contato não encontrado' })
  }

  // Extrair os novos dados do body
  const { nome, telefone, email } = req.body

  // Substituir o item na posição encontrada, mantendo o id original
  contatos[index] = { id, nome, telefone, email }

  res.json(contatos[index])
})
Por que manter o id original? O ID identifica o item — ele não deve mudar numa atualização. Por isso passamos id diretamente no novo objeto em vez de deixar vir do body. Se viesse do body, um cliente poderia enviar um ID diferente e causar inconsistência.
2
Testar o PUT no Postman

No Postman, crie uma nova requisição com método PUT. A URL deve incluir o ID de um item que existe no array. Na aba Body → raw → JSON, envie os novos dados:

{
  "nome": "Ana Paula Silva",
  "telefone": "(41) 99999-0000",
  "email": "ana.paula@email.com"
}
✓ Resultado esperado — status 200
{ "id": 1, "nome": "Ana Paula Silva", "telefone": "(41) 99999-0000", "email": "ana.paula@email.com" }
O item retornado deve ter o ID original e os dados atualizados. Confirme com um GET por ID que a alteração persiste.
findIndex retorna -1 para um ID que existe O mesmo problema do GET por ID: o ID chega como string. Confirme que const id = Number(req.params.id) está na primeira linha da rota.
3
Criar a rota DELETE /:recurso/:id
Remover um item pelo ID

O DELETE é o mais simples das quatro operações: só precisa do ID na URL. Encontra a posição com findIndex(), remove com splice() e retorna status 204 sem corpo na resposta.

O splice(index, 1) remove 1 elemento a partir da posição index, alterando o array original.

app.delete('/contatos/:id', (req, res) => {
  const id = Number(req.params.id)
  const index = contatos.findIndex(c => c.id === id)

  if (index === -1) {
    return res.status(404).json({ erro: 'Contato não encontrado' })
  }

  contatos.splice(index, 1)
  res.status(204).send()
})
Por que res.status(204).send() e não res.json()? Status 204 significa "sem conteúdo" — enviar um body junto seria contraditório. O .send() sem argumento envia uma resposta vazia, que é o comportamento correto para DELETE bem-sucedido.
4
Testar o DELETE no Postman

Crie uma requisição com método DELETE. Não precisa de body. A URL deve incluir o ID de um item existente.

✓ Resultado esperado — status 204
O painel de resposta do Postman mostra 204 No Content e nenhum body. Faça em seguida um GET na lista completa — o item deletado não deve mais aparecer.
Postman mostrando aviso ao receber 204 Algumas versões do Postman exibem um aviso para respostas sem body — isso é normal e não é um erro. O status 204 é o comportamento correto.

⭐ Quer ir além? (opcional)
  • PATCH em vez de PUT: O método PUT substitui o item inteiro — você precisa enviar todos os campos. O PATCH atualiza só os campos enviados. Implemente uma rota PATCH que só altera os campos recebidos no body, mantendo os demais intactos. Dica: contatos[index] = { ...contatos[index], ...req.body, id }.
  • Salvar Collection no Postman: Organize todas as suas requisições numa Collection com o nome do projeto. Vai ser útil ao longo de toda a UC para não precisar recriar as requisições toda aula.

REST — o padrão por trás do que você está construindo
Você tem usado GET, POST, PUT e DELETE em URLs com substantivos no plural — sem saber, está seguindo os princípios do REST (Representational State Transfer), o estilo de arquitetura mais usado para APIs web. REST define convenções como: usar substantivos nas URLs (não verbos), usar os métodos HTTP para indicar a ação, retornar o status code correto. Entender REST vai te ajudar a projetar APIs que outros desenvolvedores conseguem usar intuitivamente.
→ restfulapi.net — guia completo de REST em português e inglês
PATCH vs PUT — qual a diferença real?
PUT substitui o recurso inteiro — você envia todos os campos. PATCH atualiza parcialmente — só os campos que mudaram. Na prática, muitas APIs usam PATCH para economizar banda e simplificar o cliente. O desafio extra acima já te dá uma ideia de como implementar. A documentação do MDN tem uma explicação clara sobre quando usar cada um.
