1 As três formas de enviar dados ao servidor

Toda requisição HTTP pode carregar dados de três formas diferentes. Cada uma tem um propósito específico e fica acessível em um lugar diferente no código.

Forma Como aparece na requisição Como acessar no Express Quando usar
Route Params Na URL: /produtos/5 req.params.id Identificar um recurso específico pelo seu ID ou outro identificador único
Query Params Após ? na URL: /produtos?categoria=eletronicos req.query.categoria Filtros opcionais, buscas, ordenação — não identificam um recurso único
Body No corpo da requisição (invisível na URL) req.body.nome Criar ou atualizar um recurso — dados completos enviados no POST ou PUT
Uma analogia para fixar Imagine que você está em uma biblioteca. Pedir um livro específico pelo número de registro é como um route param — você sabe exatamente o que quer. Pedir "me mostra todos os livros de suspense disponíveis" é como um query param — você está filtrando. Preencher um formulário de cadastro de novo livro é como o body — você está enviando dados completos para criar algo novo.

2
GET /livros — lista com filtro opcional por query param
req.query — parâmetro opcional que chega após o ? na URL

Vamos criar uma rota GET que, sem nenhum parâmetro, retorna todos os livros. Se o cliente enviar ?genero=Romance, retorna apenas os livros daquele gênero.

O req.query é um objeto que contém todos os parâmetros enviados após o ? na URL. Se nenhum parâmetro for enviado, req.query é um objeto vazio {}.

app.get('/livros', (req, res) => {
  const { genero } = req.query

  // Se vier o filtro de gênero, filtra o array
  if (genero) {
    const filtrados = livros.filter(livro => livro.genero === genero)
    return res.json(filtrados)
  }

  // Sem filtro, retorna todos
  res.json(livros)
})
array.filter() percorre cada item do array e retorna um novo array contendo apenas os itens para os quais a função retornou verdadeiro. Você já viu isso no JavaScript do frontend — o comportamento é exatamente o mesmo aqui.

4
GET /livros/:id — buscar um livro específico pelo ID
req.params — parte dinâmica da URL que identifica um recurso único

O :id na rota define um parâmetro dinâmico: quando alguém acessa /livros/3, o valor 3 fica disponível em req.params.id. O nome depois dos dois pontos é livre — se você escrever :codigo, o valor fica em req.params.codigo.

Atenção importante: req.params.id sempre chega como string — mesmo que o valor enviado seja o número 3, você recebe a string "3". Para comparar com os IDs do array (que são números), você precisa converter com Number().

app.get('/livros/:id', (req, res) => {
  // Converter para número — req.params.id sempre chega como string
  const id = Number(req.params.id)

  // Procurar o livro com esse id no array
  const livro = livros.find(l => l.id === id)

  // Se não encontrou, responder com 404
  if (!livro) {
    return res.status(404).json({ erro: 'Livro não encontrado' })
  }

  res.json(livro)
})
Por que retornar 404 quando não encontra? Status 404 significa "Not Found" — o recurso não existe. Se você não fizer essa verificação e simplesmente chamar res.json(undefined), o servidor pode se comportar de forma inesperada. Sempre trate o caso "não encontrado" — é um dos erros mais comuns em APIs.
array.find() percorre o array e retorna o primeiro item para o qual a função retornou verdadeiro. Se nenhum item satisfizer a condição, retorna undefined — que é exatamente o que testamos com if (!livro).

🚀 Aplique no seu Projeto
Leve tudo que aprendeu para o index.js do seu projeto. Você vai adicionar duas novas capacidades ao seu GET principal.
1 Adicione GET /:recurso/:id ao seu projeto
Crie uma rota que recebe o ID pela URL e retorna apenas aquele item. Use Number() para converter o ID, array.find() para buscar, e retorne 404 se não encontrado. Teste no Thunder Client com um ID existente e com um ID que não existe.
2 Adicione um filtro por query param no GET principal
Pense em uma propriedade útil para filtrar no seu projeto. Exemplos: ?status=ativo para tarefas, ?categoria=bebida para produtos, ?disponivel=true para itens de biblioteca. Atualize o GET principal para suportar esse filtro opcionalmente.
3 Teste os três tipos de parâmetro no Postman
GET sem filtro (todos os itens) → GET com query param pela aba Params (filtrado) → GET com :id existente (um item) → GET com :id inexistente (404). Os quatro comportamentos devem funcionar corretamente.
⭐ Quer ir além? (opcional)
  • Dois query params ao mesmo tempo: Adicione um segundo filtro ao GET de livros — por exemplo ?disponivel=true. A URL ficaria /livros?genero=Romance&disponivel=true. Ambos os filtros precisam ser aplicados ao mesmo tempo.
  • Busca parcial com includes(): Em vez de filtrar por gênero exato, implemente uma busca no título. /livros?busca=alq deveria retornar "O Alquimista". Use livro.titulo.toLowerCase().includes(busca.toLowerCase()).
