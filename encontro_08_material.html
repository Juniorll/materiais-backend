1 Entendendo erros em APIs

O problema sem tratamento de erros Quando um erro não tratado ocorre numa rota Express, o framework entra em modo de pânico e devolve uma resposta HTML genérica com status 500 — algo como <!DOCTYPE html><html>...Error...</html>. Para uma API que deveria sempre responder em JSON, isso quebra o contrato com o cliente. Qualquer código que tente processar essa resposta como JSON vai falhar.
Erros do cliente vs erros do servidor Nem todo erro é igual. Existem dois tipos fundamentais:

— Erros do cliente (4xx): o cliente enviou algo inválido. É responsabilidade dele corrigir. Exemplos: item não encontrado (404), campos obrigatórios faltando (400), não autenticado (401).

— Erros do servidor (5xx): algo deu errado do lado do servidor — um bug no código, uma conexão com banco que falhou, uma operação inesperada. O cliente não pode fazer nada além de tentar novamente mais tarde.

Retornar o status correto não é burocracia — é comunicação precisa entre sistemas.
SituaçãoStatusTipo
Item com aquele ID não existe404Erro do cliente
Campo obrigatório faltando no body400Erro do cliente
Usuário não autenticado401Erro do cliente
Usuário sem permissão403Erro do cliente
Bug no código / exceção não prevista500Erro do servidor
Banco de dados fora do ar503Erro do servidor
O padrão: try/catch + next(err) + middleware global A estratégia que vamos adotar tem três partes:

1. try/catch em cada rota: captura qualquer erro que ocorrer dentro da rota
2. next(err) dentro do catch: envia o erro para o middleware global em vez de tratar inline
3. Middleware global de erros no index.js: um único lugar que trata todos os erros e garante resposta JSON consistente

Esse padrão é importante porque quando você conectar o banco de dados (Bloco 3), as operações SQL podem lançar erros — e o try/catch que você adiciona hoje já vai capturá-los automaticamente.

2 Manual — tornando o projeto resiliente

1
Ver o problema ao vivo — o que acontece sem tratamento
Criar uma rota de teste para ver a resposta HTML de erro

Antes de corrigir o problema, veja ele acontecendo. Adicione temporariamente uma rota de teste no seu arquivo de rotas (em routes/):

// Rota de teste — provoca erro proposital
router.get('/erro-teste', (req, res) => {
  throw new Error('Isso é um erro proposital!')
})

Faça um GET em /seuRecurso/erro-teste no Postman.

✓ Resultado esperado — o problema
Status: 500 Internal Server Error
Body: <!DOCTYPE html><html>...Error: Isso é um erro proposital!...
O Postman recebe HTML em vez de JSON. Para uma API, isso é inaceitável — qualquer sistema que consuma essa API vai falhar ao tentar processar a resposta.

Deixe essa rota por enquanto — vamos ver ela se comportar corretamente após adicionar o middleware global.

2
Criar o middleware global de erros no index.js
Um único lugar que trata todos os erros — deve vir após todas as rotas

Abra o index.js e adicione o middleware de erros após o registro de todas as rotas e antes do app.listen.

Este middleware tem quatro parâmetros — isso é obrigatório. O Express identifica um middleware de erros exatamente pela presença de quatro parâmetros: (err, req, res, next). Com três parâmetros, ele seria tratado como middleware comum e não capturaria erros.

// Middleware global de erros — deve ter 4 parâmetros exatamente
app.use((err, req, res, next) => {
  // Registra o erro no terminal para diagnóstico
  console.error(`[ERRO] ${err.message}`)

  // Usa o status do erro se definido, ou 500 como padrão
  const status = err.status || 500
  const mensagem = err.message || 'Erro interno do servidor'

  res.status(status).json({ erro: mensagem })
})
A posição importa — deve ficar após as rotas O middleware de erros só captura erros que ocorrem em rotas registradas antes dele. Se estiver antes das rotas, não capturará nada. A ordem no index.js deve ser: middlewares comuns → rotas → middleware de erros → app.listen.

Salve e teste a rota /seuRecurso/erro-teste no Postman novamente.

✓ Resultado esperado — problema resolvido
Status: 500 Internal Server Error
Body: { "erro": "Isso é um erro proposital!" }
Agora o erro é devolvido como JSON com status code correto. O terminal também mostra a linha [ERRO] Isso é um erro proposital! para ajudar no diagnóstico.
3
Adicionar try/catch nas rotas com next(err)
Capturar erros inesperados e enviá-los ao middleware global

Agora vamos envolver cada rota em try/catch. Quando ocorrer um erro inesperado, chamamos next(err) para passar ao middleware global. Veja a diferença:

Antes — sem proteção
router.get('/:id', (req, res) => {
  const id = Number(req.params.id)
  const item = itens.find(i => i.id === id)
  if (!item) {
    return res.status(404).json(
      { erro: 'Não encontrado' }
    )
  }
  res.json(item)
})
Depois — com try/catch
router.get('/:id', (req, res, next) => {
  try {
    const id = Number(req.params.id)
    const item = itens.find(i => i.id === id)
    if (!item) {
      return res.status(404).json(
        { erro: 'Não encontrado' }
      )
    }
    res.json(item)
  } catch (err) {
    next(err)
  }
})

As mudanças são mínimas: adicionar next como terceiro parâmetro da rota (necessário para chamar next(err)), envolver o código em try { } e adicionar o catch (err) { next(err) }.

Os checks de 404 que você já tinha (if (!item) return res.status(404).json(...)) continuam exatamente iguais — eles não são erros inesperados, são fluxos previstos. O try/catch só captura o que você não previu.
Por que next(err) e não res.json() no catch? Você poderia tratar o erro diretamente no catch de cada rota. Mas centralizando no middleware global, garante que todos os erros inesperados sejam tratados da mesma forma — um único lugar para ajustar, logar e formatar. Se futuramente você quiser adicionar integração com uma ferramenta de monitoramento de erros, muda só o middleware.
4
Remover a rota de teste e verificar o resultado final

Remova a rota /erro-teste que criou no Passo 1. Ela serviu para demonstrar o problema — não deve ficar no projeto.

Faça um teste final no Postman: tente buscar um item com um ID que não existe. O erro deve retornar JSON com status 404, não HTML.

✓ Resultado esperado
Status: 404 Not Found
Body: { "erro": "Não encontrado" }
JSON consistente, status correto — o servidor responde de forma previsível independente do que acontecer.

3 Teste os cenários de erro no Postman

⭐ Quer ir além? (opcional)
  • Erros com status personalizado: Ao invés de sempre retornar 500 para erros inesperados, você pode criar um erro com status específico antes de chamar next(): const err = new Error('Mensagem'); err.status = 400; next(err). Isso permite diferenciar, dentro do catch, se é um erro do cliente ou do servidor.
  • Rota 404 para caminhos não encontrados: Adicione ao final do index.js (antes do middleware de erros) uma rota curinga: app.use((req, res) => res.status(404).json({ erro: 'Rota não encontrada' })). Assim, acessar uma URL que não existe na API retorna JSON 404 em vez de um erro padrão do Express.

Classes de erro customizadas
Em vez de criar objetos de erro manualmente (const err = new Error(...); err.status = 400), projetos maiores criam classes que estendem o Error do JavaScript: class NotFoundError extends Error { constructor(msg) { super(msg); this.status = 404 } }. Isso torna o código mais expressivo: throw new NotFoundError('Produto não encontrado') é mais claro do que montar o objeto manualmente toda vez.
→ MDN — Criando tipos de erro customizados
Sentry — monitoramento de erros em produção
O middleware de log que criamos registra erros no terminal — útil no desenvolvimento. Em produção, o terminal não é monitorado 24/7. O Sentry é uma ferramenta de monitoramento de erros: toda vez que um erro ocorre em produção, ele captura automaticamente — com stack trace completo, dados do request e contexto — e envia uma notificação. É o padrão da indústria para monitoramento de erros em aplicações web.
