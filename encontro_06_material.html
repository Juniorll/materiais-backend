1 O que é middleware

Antes de qualquer código novo, é importante entender o conceito — porque middleware aparece em todo lugar a partir daqui.

A definição Middleware é uma função que fica no meio do caminho entre a requisição chegar e a rota responder. O nome vem do inglês: middle (meio) + ware (software). Toda vez que uma requisição entra no servidor Express, ela passa por uma fila de funções antes de chegar na rota — cada uma dessas funções é um middleware.

Você já usa middleware desde o Encontro 03 A linha app.use(express.json()) que você adicionou no E03 — essa é exatamente um middleware. O express.json() é uma função que intercepta a requisição, lê o corpo JSON, coloca o resultado em req.body e passa para o próximo. Você não precisou escrever essa lógica porque ela já vem pronta no Express — mas o mecanismo é o mesmo que você vai usar para criar os seus próprios middlewares hoje.
A assinatura de um middleware Todo middleware tem exatamente três parâmetros:

(req, res, next) => { ... }

— req: o objeto da requisição (o mesmo que nas rotas)
— res: o objeto da resposta (o mesmo que nas rotas)
— next: uma função que você chama para passar para o próximo da fila

A diferença de uma rota: um middleware não precisa enviar uma resposta — ele faz seu trabalho e chama next() para continuar. Se não chamar next(), a requisição para ali e nunca chega na rota.

2 Manual — criando middleware no projeto

Tudo que segue vai diretamente no index.js do seu projeto — sem arquivos separados.

1
Criar o middleware de log
Registra método, rota e horário de cada requisição no terminal

Abra o index.js e adicione o código abaixo após o app.use(express.json()) e antes das rotas. A posição importa: middleware registrado com app.use() afeta apenas as rotas que vêm depois dele no arquivo.

// Middleware de log — registra toda requisição recebida
app.use((req, res, next) => {
  const horario = new Date().toLocaleTimeString('pt-BR')
  console.log(`[${horario}] ${req.method} ${req.path}`)
  next()
})

Salve o arquivo. O nodemon reinicia o servidor automaticamente.

O next() na última linha é obrigatório. Ele diz ao Express: "este middleware terminou, pode passar para o próximo da fila". Sem ele, a requisição fica presa aqui e nenhuma rota vai responder.
2
Observar o middleware funcionando no terminal
O log aparece em tempo real a cada requisição

Com o servidor rodando, faça algumas requisições no Postman — qualquer rota, qualquer método. Observe o terminal do VS Code.

✓ O que aparece no terminal
[14:32:05] GET /produtos
[14:32:11] POST /produtos
[14:32:18] GET /produtos/3
[14:32:24] DELETE /produtos/3
Cada requisição aparece no terminal com o horário exato, o método e o caminho. O servidor continua respondendo normalmente — o middleware só adicionou o log, não interferiu nas rotas.
Por que isso é útil? Durante o desenvolvimento, o log te ajuda a entender o que está acontecendo no servidor em tempo real — quantas requisições chegaram, em que ordem, quais rotas estão sendo chamadas. Em produção, logs são essenciais para diagnóstico de problemas.
3
Ver o que acontece sem o next()
O erro mais comum com middleware — entender para nunca cometer

Remova temporariamente o next() do middleware (só para ver o efeito — vai desfazer em seguida):

// ⚠ VERSÃO COM BUG — não deixe assim
app.use((req, res, next) => {
  const horario = new Date().toLocaleTimeString('pt-BR')
  console.log(`[${horario}] ${req.method} ${req.path}`)
  // next() removido
})

Salve e tente fazer qualquer requisição no Postman.

✓ Resultado esperado — requisição trava
O Postman fica esperando indefinidamente — eventualmente dá timeout. O log aparece no terminal (o middleware rodou), mas a requisição nunca chegou na rota porque next() não foi chamado.
Requisição travada / timeout no Postman O middleware não está chamando next(). Toda vez que uma requisição "some" sem resposta, verifique se todos os middlewares no caminho estão chamando next() ao final.

Restaure o next() no middleware antes de continuar.

4
Adicionar dados ao req dentro do middleware
Middleware pode enriquecer a requisição antes de chegar na rota

Middlewares não só observam as requisições — eles podem modificar o objeto req para adicionar informações que as rotas podem usar. Isso é exatamente o que o middleware de autenticação vai fazer no Bloco 4: ele vai verificar o token e colocar os dados do usuário logado em req.usuario.

Para entender o mecanismo agora, adicione o horário da requisição ao objeto req. Atualize o middleware de log para incluir essa linha:

app.use((req, res, next) => {
  const horario = new Date().toLocaleTimeString('pt-BR')
  console.log(`[${horario}] ${req.method} ${req.path}`)
  req.horario = horario  // disponível em todas as rotas
  next()
})

Agora qualquer rota pode acessar req.horario. Por exemplo, adicione temporariamente a uma rota GET para ver funcionando:

app.get('/', (req, res) => {
  res.json({
    status: 'online',
    horario: req.horario  // veio do middleware
  })
})
✓ GET / no Postman
{ "status": "online", "horario": "14:45:22" }
O horário foi definido no middleware e chegou na rota via req. Este é o mesmo mecanismo que o middleware de autenticação usará para passar req.usuario para as rotas protegidas.
🚀 Aplique no seu Projeto
Você já fez os passos acima no seu projeto. Agora refine e complemente.
1 Confirme a posição do middleware no arquivo
O middleware de log deve estar depois de app.use(express.json()) e antes de todas as suas rotas. Abra o index.js e verifique a ordem. Se estiver após as rotas, o log não vai aparecer — mova para antes.
2 Personalize a mensagem de log do projeto
Adapte o log para incluir o nome do projeto. Exemplo: [14:32:05] catalogo-filmes | GET /filmes. Substitua "catalogo-filmes" pelo nome do seu projeto. Pequeno detalhe, mas torna o terminal muito mais legível quando você tem múltiplos projetos abertos.
3 Faça o checklist de rotas no Postman e observe o terminal
Execute GET, POST, PUT e DELETE no Postman. Para cada requisição, confirme que uma linha aparece no terminal com método e rota corretos. Ao final, o terminal deve mostrar um histórico de tudo que foi testado.
⭐ Quer ir além? (opcional)
  • Logar o status da resposta: O middleware atual só sabe o método e o rota quando a requisição chega. Para logar também o status da resposta, use o evento res.on('finish', ...) dentro do middleware: res.on('finish', () => console.log(res.statusCode)). Esse evento dispara depois que a resposta foi enviada.
  • Middleware para rota específica: Até agora usamos app.use() que aplica o middleware a todas as rotas. Você pode aplicar apenas a uma rota passando o middleware como segundo argumento: app.get('/admin', meuMiddleware, (req, res) => {...}). Teste essa variação.

morgan — o middleware de log padrão do mercado
O middleware de log que você criou hoje faz o essencial. O morgan é um pacote npm que faz a mesma coisa, mas com muito mais opções: formatos pré-definidos, cores no terminal, registro em arquivo, filtros por status code. Em empresas, usar morgan (ou similar) é mais comum do que criar um middleware de log do zero. Vale conhecer para quando você ver em projetos reais.
→ github.com/expressjs/morgan — documentação e exemplos
helmet — segurança com uma linha
O helmet é um pacote npm que funciona como middleware e adiciona dezenas de cabeçalhos de segurança HTTP ao servidor automaticamente — protegendo contra ataques comuns como clickjacking, XSS e injeção de conteúdo. É usado em praticamente todo projeto Express em produção. O uso é simplesmente: app.use(helmet()).
→ helmetjs.github.io — documentação completa
