1 Revisar o MVP — que dados o sistema precisa guardar?

Responda estas perguntas sobre o seu projeto antes de abrir qualquer arquivo:

  • Quais são as entidades principais do sistema? (cada entidade vai virar uma tabela)
  • O que cada entidade precisa guardar? (cada dado vai virar uma coluna)
  • Quais entidades se relacionam? Quem é pai, quem é filho?
  • Existe algum dado que você está repetindo em mais de uma tabela? (sinal para criar uma nova entidade)
  • O que o sistema precisa listar, filtrar ou cruzar? (indica quais JOINs você vai precisar)
A tabela de usuários não entra agora A autenticação e a tabela de usuários serão adicionadas nos encontros 20 a 24. Modele tudo que o sistema precisa exceto a tabela de usuários — ela vem pronta com o módulo de JWT.

db.pragma('foreign_keys = ON')

5 Validar com dados de teste

4
Inserir pelo menos um registro em cada tabela e confirmar no DB Browser

Inicie o servidor (npm run dev). Para cada tabela do banco, faça um POST no Postman com dados de teste. A ordem importa: inserir primeiro nas tabelas pai antes das filhas.

Após cada inserção, abra o DB Browser, clique em Refresh e confirme que o registro aparece na tabela correta. Ao final, todas as tabelas devem ter pelo menos um registro com os campos corretos preenchidos.

Rotas que ainda não existem Se alguma tabela nova ainda não tem rotas, insira os dados de teste diretamente pelo DB Browser — clique em Browse Data, selecione a tabela e use o botão de inserção. O objetivo agora é validar a estrutura, não as rotas.
⭐ Quer ir além? (opcional)
1 DEFAULT com funções de data do SQLite

O SQLite tem funções nativas para data e hora que você pode usar como valor padrão de uma coluna:

-- Data atual no formato AAAA-MM-DD
data TEXT DEFAULT (date('now'))

-- Data e hora atual no formato AAAA-MM-DD HH:MM:SS
criado_em TEXT DEFAULT (datetime('now'))

-- Timestamp Unix (segundos desde 1970)
timestamp INTEGER DEFAULT (strftime('%s','now'))

Use datetime('now') em tabelas onde o horário da inserção precisa ser registrado automaticamente, sem depender do JavaScript para enviar o valor.

2 Restrição CHECK — validar valores diretamente no banco

Além de NOT NULL e UNIQUE, o SQLite suporta a restrição CHECK para validar valores antes de aceitar a inserção:

-- nota só aceita valores entre 1 e 5
nota INTEGER NOT NULL CHECK (nota >= 1 AND nota <= 5)

-- status só aceita valores específicos
status TEXT DEFAULT 'pendente'
  CHECK (status IN ('pendente','confirmado','cancelado'))

-- preco não pode ser negativo
preco REAL NOT NULL CHECK (preco >= 0)

Isso cria uma segunda camada de proteção além da validação nas rotas.
