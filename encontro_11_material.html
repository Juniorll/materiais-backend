1 Por que banco de dados existe

✗ Array em memória (até aqui)
  • Dados vivem na RAM do processo Node.js
  • Somem quando o servidor reinicia
  • Limitados pela memória disponível
  • Sem garantias de consistência
  • Bom para aprender, ruim para produção
✓ Banco de dados (a partir de hoje)
  • Dados gravados em arquivo no disco
  • Sobrevivem a reinícios e quedas
  • Suportam milhões de registros
  • Estrutura e tipos bem definidos
  • Padrão real de mercado
O que é um banco de dados relacional Um banco de dados relacional organiza informações em tabelas — parecidas com planilhas. Cada tabela tem colunas (os atributos, como nome, preço, categoria) e linhas (os registros individuais, cada produto, cada usuário). A palavra "relacional" vem da capacidade de relacionar tabelas entre si — algo que você vai explorar no E16, mais adiante no bloco.
O que é o SQLite e por que começamos com ele SQLite é um banco de dados relacional que guarda tudo em um único arquivo no seu projeto — sem precisar instalar e configurar um servidor de banco separado (como acontece com PostgreSQL ou MySQL). Isso o torna ideal para aprender: você foca na lógica de tabelas e consultas SQL sem a complexidade de administrar um servidor. É usado em produção em milhões de aplicativos — inclusive no seu navegador e no seu celular, provavelmente agora mesmo.

2 Manual — criando o primeiro banco

1
Instalar o better-sqlite3
A biblioteca que conecta o Node.js ao arquivo do banco SQLite

O better-sqlite3 é o pacote que permite ao Node.js ler e escrever no arquivo do banco SQLite. É chamado de "better" (melhor) porque é mais rápido e simples de usar que alternativas mais antigas.

npm install better-sqlite3
✓ Resultado esperado
added 3 packages in 4s
Instalação pode demorar um pouco mais Esse pacote compila código nativo durante a instalação — pode levar de 10 a 30 segundos, mais que pacotes comuns. É normal.
2
Criar o módulo de conexão com o banco
Um arquivo central que vai abrir e configurar o banco

Crie um arquivo chamado database.js na raiz do projeto. Ele será o ponto único de conexão com o banco — todo o resto do projeto vai importar a conexão a partir daqui.

const Database = require('better-sqlite3')

// Abre (ou cria, se não existir) o arquivo do banco
const db = new Database('banco.db')

module.exports = db

Quando esse código roda pela primeira vez, o arquivo banco.db é criado automaticamente na raiz do projeto — vazio, sem tabelas ainda.

Diferente do .env, o arquivo banco.db normalmente não vai para o GitHub — ele contém dados que mudam constantemente e pode crescer muito. Vamos adicioná-lo ao .gitignore no próximo passo.
3
Adicionar o banco.db ao .gitignore

Abra o .gitignore e adicione:

node_modules/
.env
banco.db
Por que isso importa Sem isso, cada vez que você ou um colega rodar o projeto localmente, o banco de dados ficaria sujeito a conflitos no Git — como dois arquivos binários diferentes tentando ser mesclados. Cada ambiente (sua máquina, a do colega, produção) deve ter seu próprio arquivo de banco.
4
Criar a primeira tabela
CREATE TABLE — definindo a estrutura dos dados

Vamos criar a tabela do recurso principal do seu projeto. O comando SQL CREATE TABLE define o nome da tabela e suas colunas, cada uma com um tipo de dado.

Adicione ao database.js, após a criação da conexão:

const Database = require('better-sqlite3')
const db = new Database('banco.db')

// Cria a tabela se ela ainda não existir
db.exec(`
  CREATE TABLE IF NOT EXISTS produtos (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    nome TEXT NOT NULL,
    preco REAL NOT NULL
  )
`)

module.exports = db

Substitua produtos e as colunas pelo nome e atributos do recurso do seu projeto.

Tipo SQLiteEquivalente em JSExemplo de uso
INTEGERnumberid, quantidade, idade
REALnumber (decimal)preço, nota, peso
TEXTstringnome, descrição, e-mail
Decifrando a linha do id id INTEGER PRIMARY KEY AUTOINCREMENT — PRIMARY KEY marca essa coluna como o identificador único de cada linha (o SQLite cria automaticamente um índice rápido para buscas por ela). AUTOINCREMENT faz o banco gerar o próximo número sozinho a cada inserção — você nunca mais vai precisar calcular o próximo ID manualmente como fazia com filmes.length + 1.
NOT NULL significa que aquela coluna é obrigatória — o banco rejeita a inserção se ela vier vazia. É uma primeira camada de proteção dos dados, antes mesmo da validação que vamos reforçar no E18.
5
Importar o banco no index.js e testar

No index.js, adicione o import do banco logo após os outros requires (isso garante que a tabela seja criada assim que o servidor iniciar):

require('dotenv').config()
const express = require('express')
const db = require('./database') // ← garante que o banco e a tabela existem
const app = express()

Salve e observe o terminal. O nodemon reinicia. Se nenhum erro aparecer, o banco e a tabela foram criados com sucesso.

✓ Resultado esperado
Um novo arquivo banco.db aparece na raiz do projeto, na barra lateral do VS Code. O servidor continua rodando normalmente — nenhuma rota muda de comportamento ainda, porque elas continuam usando o array em memória. Isso muda no próximo encontro.
"Cannot find module 'better-sqlite3'" O pacote não foi instalado corretamente. Rode npm install better-sqlite3 novamente.
Erro de sintaxe SQL ao iniciar Revise o texto entre os acentos graves (`...`) no db.exec(). Vírgulas faltando ou sobrando entre as colunas são o erro mais comum.
6
Instalar e abrir o DB Browser for SQLite
Interface visual para ver o que está dentro do banco

O DB Browser é um programa com interface gráfica que permite abrir um arquivo .db e ver suas tabelas, colunas e dados — sem escrever nenhum SQL. É excelente para conferir visualmente se o que o código está fazendo bate com o esperado.

  1. Baixe o DB Browser for SQLite em sqlitebrowser.org (gratuito, disponível para Windows, Mac e Linux)
  2. Instale no seu computador local — esta ferramenta roda fora do Codespace, direto na sua máquina
  3. Para abrir o arquivo do banco que está no Codespace, baixe-o: clique com botão direito em banco.db na barra lateral do VS Code → Download
  4. Abra o DB Browser, clique em Open Database e selecione o arquivo baixado
✓ Resultado esperado
Na aba Database Structure, você vê a tabela que criou, com o nome e as colunas exatamente como definiu no CREATE TABLE. Na aba Browse Data, a tabela aparece vazia — ainda não inserimos nenhum registro.
O arquivo baixado é uma cópia estática Toda vez que você quiser ver o estado mais atual do banco depois de testar inserções pelo Postman, vai precisar baixar o arquivo de novo. No próximo encontro vamos trabalhar com isso de forma mais prática.
7
Inserir um dado manualmente pelo DB Browser
Ver o fluxo completo: inserir, salvar, conferir

Na aba Browse Data do DB Browser, clique em Insert new record (ícone de +). Preencha os campos manualmente — exceto o id, que o AUTOINCREMENT preenche sozinho.

Clique em Write Changes (importante: o DB Browser não salva automaticamente).

✓ Resultado esperado
O registro aparece na lista da aba Browse Data, com um id atribuído automaticamente (1, se for o primeiro).
Esse dado existe apenas no arquivo que você baixou — ele não está sincronizado de volta com o Codespace. Essa limitação também é resolvida no próximo encontro, quando passamos a inserir dados via código, direto no Codespace.

⭐ Quer ir além? (opcional)

Se você já terminou e quer aprofundar antes do próximo encontro, estas duas explorações ajudam a fixar o que aprendeu hoje.

1 Adicione uma coluna de data de criação

É comum que tabelas guardem quando cada registro foi criado. O SQLite tem uma função especial, CURRENT_TIMESTAMP, que preenche automaticamente a data e hora atual quando nenhum valor é fornecido.

Como fazer:

  1. No CREATE TABLE, adicione uma nova coluna: criado_em TEXT DEFAULT CURRENT_TIMESTAMP
  2. Apague o arquivo banco.db existente (já que CREATE TABLE IF NOT EXISTS não altera tabelas já criadas) e reinicie o servidor para recriá-lo com a nova coluna
  3. Insira um novo registro pelo DB Browser deixando o campo criado_em vazio — observe que o SQLite preenche sozinho
db.exec(`
  CREATE TABLE IF NOT EXISTS produtos (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    nome TEXT NOT NULL,
    preco REAL NOT NULL,
    criado_em TEXT DEFAULT CURRENT_TIMESTAMP
  )
`)
2 Explore o SQL Editor do DB Browser

Além da interface visual de Browse Data, o DB Browser tem uma aba Execute SQL, onde você pode escrever comandos SQL diretamente — uma prévia do que vamos fazer pelo código no próximo encontro.

Como fazer:

  1. Abra a aba Execute SQL no DB Browser
  2. Digite SELECT * FROM produtos; e clique no botão de Play (▶) ou pressione Ctrl+Enter
  3. O resultado aparece numa tabela abaixo — é a mesma consulta que o código vai fazer no E12, só que digitada manualmente

Experimente também SELECT * FROM produtos WHERE preco > 50; para ver uma busca filtrada — você vai escrever exatamente esse tipo de comando dentro do Node.js em breve.

PostgreSQL e a evolução natural do seu projeto
SQLite é perfeito para aprender e para projetos pequenos, mas tem limitações em produção com múltiplos usuários simultâneos. O PostgreSQL é um banco de dados relacional muito mais robusto — usado por empresas de todos os tamanhos — que roda como um servidor separado (não um único arquivo). A boa notícia: a lógica SQL que você está aprendendo agora é quase idêntica entre os dois. No fim da UC, quando o projeto for para produção no Railway, a migração de SQLite para PostgreSQL é uma troca pequena de configuração, não uma reescrita do projeto.
→ postgresql.org/about
ORMs — escrever menos SQL com Prisma ou Sequelize
Um ORM (Object-Relational Mapping) é uma camada que traduz objetos JavaScript em comandos SQL automaticamente — você escreve produto.create({ nome: 'X', preco: 10 }) em vez de escrever o INSERT manualmente. Ferramentas como Prisma e Sequelize são populares no mercado Node.js. Nesta UC escrevemos SQL diretamente de propósito — entender o que está por baixo do capô facilita muito aprender um ORM depois, e evita depender de uma "caixa-preta" sem saber o que ela faz.
→ prisma.io
