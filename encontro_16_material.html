1 Por que tabelas se relacionam?

O problema da repetição de dados Se você guardar o nome do cliente em cada linha da tabela de pedidos, qualquer mudança no nome exige atualizar dezenas de registros. A solução: o nome do cliente fica em uma tabela clientes, e cada pedido guarda apenas o id do cliente. Quando precisar do nome, busca na tabela certa. Isso é a base do modelo relacional — cada dado em um único lugar.
Chave Estrangeira (FK) — o elo entre as tabelas É uma coluna na tabela filha que guarda o id de um registro da tabela pai. Ela aponta para o pai. A tabela itens_pedido tem uma coluna pedido_id: esse valor diz a qual pedido cada item pertence. Sem a FK, os dados ficam soltos — não há como saber quem pertence a quem.
Relacionamento 1:N — um pai, vários filhos Um pedido tem vários itens. Um cliente tem vários agendamentos. Um livro tem vários empréstimos. Esse é o relacionamento mais comum em sistemas reais e o que você vai implementar hoje.

2 Criar a tabela filha no banco do projeto

1
Adicionar foreign_keys e a tabela filha ao database.js
Localize o seu projeto — a tabela pai que já existe não muda

Adicione db.pragma('foreign_keys = ON') logo após criar a instância do banco. Em seguida, adicione o novo db.exec() com a tabela filha do seu projeto.

🛒 Sistema de Pedidos / Loja

Tabela pai: pedidos  →  Nova tabela filha: itens_pedido

db.pragma('foreign_keys = ON')

db.exec(`CREATE TABLE IF NOT EXISTS itens_pedido (

  produto TEXT NOT NULL,
  quantidade INTEGER NOT NULL,
  preco REAL NOT NULL,
  pedido_id INTEGER NOT NULL,
  FOREIGN KEY (pedido_id) REFERENCES pedidos(id)
)`)

Por que foreign_keys = ON é necessário O SQLite não verifica FKs por padrão — sem esse pragma você consegue inserir um filho com id de pai inexistente sem nenhum erro. O pragma deve vir imediatamente após o new Database(...).
A tabela existente não precisa ser alterada Só o novo db.exec() com a tabela filha é adicionado. O servidor vai criá-la automaticamente no próximo start — sem apagar o banco.db.

GET /:id/filhos — listar registros filhos de um pai
Substitua PAI, FILHOS e CAMPO_FK pelos nomes da tabela acima
// GET /pai/:id/filhos
router.get('/:id/filhos', (req, res, next) => {
  try {
    const pai = db.prepare('SELECT * FROM PAI WHERE id = ?').get(req.params.id)
    if (!pai) return res.status(404).json({ erro: 'Não encontrado' })

    const filhos = db.prepare('SELECT * FROM FILHOS WHERE CAMPO_FK = ?')
      .all(req.params.id)
    res.json(filhos)
  } catch (err) { next(err) }
})
5
POST /:id/filhos — criar um filho vinculado ao pai
Sempre verificar se o pai existe antes de inserir
// POST /pai/:id/filhos
router.post('/:id/filhos', (req, res, next) => {
  try {
    const pai = db.prepare('SELECT * FROM PAI WHERE id = ?').get(req.params.id)
    if (!pai) return res.status(404).json({ erro: 'Não encontrado' })

    const { campo1, campo2 } = req.body
    if (!campo1) return res.status(400).json({ erro: 'campo1 é obrigatório' })

    const resultado = db.prepare(
      'INSERT INTO FILHOS (campo1, campo2, CAMPO_FK) VALUES (?, ?, ?)'
    ).run(campo1, campo2, req.params.id)

    const novo = db.prepare('SELECT * FROM FILHOS WHERE id = ?')
      .get(resultado.lastInsertRowid)
    res.status(201).json(novo)
  } catch (err) { next(err) }
})
Por que verificar o pai antes de inserir o filho? Retornar "404 — Pedido não encontrado" é muito melhor do que deixar o banco lançar um erro interno de FK. Quem usa a API entende o 404 — não entende uma exceção do SQLite.

4 JOIN — retornar o pai com seus filhos em uma só resposta

O que é um JOIN? Em vez de duas queries separadas (buscar o pai, depois buscar os filhos), o JOIN faz tudo em uma query — o banco entrega os dados das duas tabelas combinados. O resultado é mais eficiente e mais expressivo para quem consome a API.
7
Atualizar GET /:id para retornar o pai com seus filhos
Deve vir ANTES das rotas /:id/filhos no arquivo

Substitua ou atualize a rota GET /:id existente. A posição no arquivo importa: ela deve aparecer antes das rotas /:id/filhos, senão o Express vai confundir os caminhos.

// GET /pai/:id — retorna o pai com seus filhos via JOIN
// Posicionar ANTES de /:id/filhos no arquivo
router.get('/:id', (req, res, next) => {
  try {
    const pai = db.prepare('SELECT * FROM PAI WHERE id = ?').get(req.params.id)
    if (!pai) return res.status(404).json({ erro: 'Não encontrado' })

    const filhos = db.prepare(`
      SELECT FILHOS.*
      FROM FILHOS
      INNER JOIN PAI ON FILHOS.CAMPO_FK = PAI.id
      WHERE PAI.id = ?
    `).all(req.params.id)

    res.json({ ...pai, filhos })
  } catch (err) { next(err) }
})

Teste com GET /pai/1 — a resposta deve trazer todos os campos do pai mais uma chave filhos com array dos registros vinculados.

Como ler o INNER JOIN "Traga linhas de FILHOS unindo com PAI onde o CAMPO_FK do filho bate com o id do pai." O WHERE PAI.id = ? filtra para um pai específico. O { ...pai, filhos } combina os campos do pai com o array de filhos em um único objeto JSON.
⭐ Quer ir além? (opcional)
1 LEFT JOIN — incluir pais sem filhos no resultado

O INNER JOIN só retorna pais que têm ao menos um filho. O LEFT JOIN inclui todos os pais — os sem filhos aparecem com array vazio. Crie uma rota GET /pai/resumo (antes de GET /:id no arquivo) com LEFT JOIN e COUNT:

SELECT PAI.id, PAI.nome, COUNT(FILHOS.id) AS total
FROM PAI
LEFT JOIN FILHOS ON FILHOS.CAMPO_FK = PAI.id
GROUP BY PAI.id
2 ON DELETE CASCADE — excluir filhos automaticamente com o pai

Com foreign_keys = ON, deletar um pai que tem filhos falha — o banco protege a integridade. Adicione ON DELETE CASCADE na FK para que os filhos sejam removidos automaticamente quando o pai for deletado:

FOREIGN KEY (CAMPO_FK) REFERENCES PAI(id) ON DELETE CASCADE

Atenção: para aplicar em tabela já criada, apague o banco.db e reinicie o servidor.

Relacionamentos N:M — muitos para muitos
Um produto pode pertencer a várias categorias; uma categoria pode ter vários produtos. Isso é N:M — não se resolve com uma FK simples. A solução é uma tabela de junção com duas FKs, uma para cada lado. Aparece em sistemas com tags, permissões e categorias múltiplas.
