1 Entendendo o que vamos construir

Antes de escrever qualquer código, é importante entender o que cada ferramenta faz e por que você vai usá-la. Leia com atenção — fará tudo mais fácil.

O que é um servidor? Pense em um servidor como um garçom de restaurante. O garçom fica esperando alguém fazer um pedido. Quando o pedido chega ("me traz um suco"), ele processa e devolve uma resposta ("aqui está seu suco"). O servidor web faz exatamente isso: fica esperando requisições chegarem pela internet, processa cada uma e devolve uma resposta. A diferença é que em vez de pedidos de comida, são requisições HTTP — e em vez de pratos, são dados em formato JSON ou páginas HTML.
O que é Node.js? Você já conhece JavaScript — a linguagem que roda nos navegadores para deixar as páginas interativas. O Node.js é uma forma de rodar JavaScript fora do navegador, diretamente no computador ou em um servidor. Isso significa que você pode usar a linguagem que já conhece para criar o backend da aplicação. Não precisa aprender uma linguagem nova.
O que é Express? Node.js puro consegue criar servidores, mas exige muito código repetitivo. Express é um framework — uma ferramenta construída em cima do Node.js que simplifica esse trabalho. Com Express, criar uma rota que responde a uma requisição levam 3 linhas de código. Sem ele, levariam 20. Express é um dos frameworks mais populares do mundo e está em uso em milhares de empresas.
O que é npm? npm significa Node Package Manager — o gerenciador de pacotes do Node.js. É como uma loja de ferramentas online. Quando você quer adicionar uma funcionalidade ao projeto (como o Express, por exemplo), em vez de escrever tudo do zero, você usa o npm para baixar e instalar o que precisa. Ele também controla quais ferramentas seu projeto usa e em quais versões.
Por que vamos usar GitHub Codespaces? Os computadores desta sala têm um sistema de proteção que apaga tudo que foi instalado quando você desliga. Isso significa que, sem uma solução, você perderia todo o progresso a cada aula. O GitHub Codespaces resolve isso: é um ambiente de desenvolvimento que roda na nuvem, dentro do próprio VS Code. Seu código fica salvo no GitHub — não na máquina local. Você pode entrar de qualquer computador e continuar de onde parou.

2 Configurando o ambiente

Esta seção só precisa ser feita uma vez. Nos próximos encontros você pula direto para abrir o Codespace e continuar o projeto.

1
Verificar conta no GitHub
github.com

Acesse github.com no navegador. Se você já criou sua conta, faça login. Se ainda não tem conta, clique em Sign up e crie agora — é gratuito.

Guarde bem o seu usuário e senha do GitHub. Você vai usar toda aula.

Por que o GitHub? GitHub é onde a maioria das empresas de tecnologia armazena seus projetos. Ao final da UC você vai ter um portfólio real lá, com todo o histórico de tudo que construiu.
2
Criar o repositório do seu projeto no GitHub
Um repositório é a "pasta" do projeto no GitHub

Um repositório (ou "repo") é onde o seu projeto fica armazenado no GitHub — com todo o histórico de alterações. Vamos criar o repositório do seu projeto integrador agora.

  1. No GitHub, clique no ícone + (canto superior direito) → New repository
  2. Em Repository name, coloque o nome do seu projeto em letras minúsculas com hífens. Ex: minha-agenda, cardapio-digital, sistema-pedidos
  3. Em Description, escreva uma linha sobre o projeto (ex: "API backend do meu projeto integrador")
  4. Deixe marcado como Public
  5. Marque a opção Add a README file
  6. Clique em Create repository
✓ Resultado esperado
Você será redirecionado para a página do seu repositório. Ele estará vazio, com apenas o arquivo README.md.
3
Configurar o Settings Sync no VS Code
Suas extensões serão reinstaladas automaticamente toda aula

O VS Code tem um recurso chamado Settings Sync: ele salva suas configurações e extensões na sua conta do GitHub. Assim, toda vez que você abrir o VS Code e fizer login, tudo volta automaticamente — mesmo que o computador tenha sido resetado.

  1. Abra o VS Code
  2. No canto inferior esquerdo, clique no ícone de pessoa (perfil)
  3. Clique em Turn on Settings Sync…
  4. Uma janela vai perguntar o que sincronizar — deixe tudo marcado e clique em Sign in & Turn on
  5. Escolha Sign in with GitHub
  6. O navegador vai abrir pedindo autorização — clique em Authorize
  7. Volte ao VS Code — a sincronização começa automaticamente
✓ Resultado esperado
No canto inferior esquerdo do VS Code, onde estava o ícone de pessoa, agora aparece o seu nome de usuário do GitHub. As extensões que você já tinha instaladas anteriormente começam a ser instaladas automaticamente (pode levar 1-2 minutos).
Para as próximas aulas: Basta abrir o VS Code e fazer login com o GitHub quando solicitado. Em cerca de 2 minutos tudo estará pronto — sem precisar instalar nada manualmente.
4
Abrir o projeto no GitHub Codespaces
Seu ambiente de desenvolvimento na nuvem — imune ao reset da máquina

Agora vamos abrir o seu repositório dentro do Codespaces. A primeira vez demora cerca de 1-2 minutos porque o ambiente está sendo criado. Nas próximas vezes é quase instantâneo.

  1. No GitHub, acesse o seu repositório
  2. Clique no botão verde Code
  3. Clique na aba Codespaces
  4. Clique em Create codespace on main
  5. Aguarde o ambiente carregar (uma tela de carregamento aparece)
✓ Resultado esperado
O VS Code abre — seja no navegador ou no VS Code do computador (dependendo da configuração). Você vai ver a estrutura de pastas do repositório à esquerda e um terminal na parte de baixo.
VS Code local ou no navegador? Se o VS Code desktop abrir e perguntar se quer abrir o Codespace localmente, clique em Open in VS Code Desktop. Se abrir no navegador, funciona da mesma forma — a experiência é idêntica.
Próximas aulas — como abrir o projeto Vá em github.com → seu repositório → Code → Codespaces → clique no codespace já existente (não crie um novo). Assim você abre exatamente de onde parou.
5
Verificar que o Node.js está disponível
O Codespaces já vem com Node.js instalado

O Codespaces já vem com o Node.js instalado. Vamos confirmar isso no terminal.

No VS Code, clique no menu Terminal → New Terminal (ou use o atalho Ctrl + `). Um terminal aparece na parte inferior. Digite:

node --version
npm --version
✓ Resultado esperado
v20.x.x
10.x.x
Os números exatos podem variar — o importante é que apareçam versões, sem mensagens de erro.
Erro: "command not found: node" Se isso aparecer, chame o professor. O Codespaces deve ter Node.js por padrão, mas ocasionalmente pode ser necessário instalar manualmente.

3 Inicializando o projeto Node.js

Com o ambiente pronto, vamos inicializar o projeto. Todo projeto Node.js começa com os mesmos passos.

6
Inicializar o projeto com npm init
Cria o package.json — a "identidade" do projeto

Todo projeto Node.js tem um arquivo chamado package.json. Esse arquivo é como a carteira de identidade do projeto: guarda o nome, a versão, quais ferramentas ele usa (chamadas de dependências) e quais comandos podem ser executados.

No terminal, digite o comando abaixo. A flag -y significa "aceitar tudo com os valores padrão" — sem precisar responder perguntas.

npm init -y
✓ Resultado esperado
Wrote to /workspaces/seu-projeto/package.json:
{
  "name": "seu-projeto",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  ...
}
Um arquivo package.json aparece na barra lateral esquerda do VS Code.
Curiosidade: Abra o package.json clicando nele na barra lateral. Você pode ver o nome do projeto, a versão e outras informações. Ao longo da UC esse arquivo vai crescer à medida que adicionamos ferramentas.
7
Instalar o Express
O framework que vai simplificar a criação do servidor

Para instalar o Express, usamos o npm. O comando npm install busca o pacote na internet, baixa e salva na pasta do projeto.

npm install express
✓ Resultado esperado
added 64 packages in 3s

Duas coisas novas aparecem no projeto:

📁 node_modules/ ← pasta com o Express e suas dependências (não edite!) 📄 package.json ← atualizado com Express listado em "dependencies" 📄 package-lock.json ← criado automaticamente — não apague
O que é a pasta node_modules? É onde o Express (e tudo que ele precisa para funcionar) fica salvo. Essa pasta pode ter centenas de arquivos — é normal. Você nunca precisa mexer nela diretamente.
Não suba a pasta node_modules para o GitHub! Ela é grande e desnecessária — o GitHub só precisa do package.json. Vamos resolver isso logo mais com o .gitignore.
8
Instalar o nodemon
Reinicia o servidor automaticamente quando você salva o código

Toda vez que você alterar um arquivo, o servidor precisa ser reiniciado para aplicar as mudanças. Fazer isso manualmente (Ctrl+C, npm start, repetir) seria muito trabalhoso. O nodemon monitora seus arquivos e reinicia o servidor sozinho toda vez que você salva.

Note a flag --save-dev: isso diz ao npm que o nodemon é uma ferramenta de desenvolvimento — só usada durante a programação, não no servidor em produção.

npm install nodemon --save-dev
✓ Resultado esperado
added 28 packages in 2s
Abra o package.json — agora você verá duas seções: "dependencies" com o Express, e "devDependencies" com o nodemon.
9
Criar o .gitignore
Dizer ao Git o que ele não deve enviar para o GitHub

O arquivo .gitignore lista arquivos e pastas que o Git deve ignorar — ou seja, que não devem ir para o GitHub. A pasta node_modules é o exemplo clássico: ela é gerada automaticamente pelo npm install e não precisa estar no repositório.

Na barra lateral do VS Code, clique com o botão direito → New File e crie um arquivo chamado exatamente .gitignore (com o ponto na frente). Dentro dele, escreva:

node_modules/

Salve o arquivo com Ctrl + S.

✓ Resultado esperado
O arquivo .gitignore aparece na barra lateral. A pasta node_modules pode ficar com a cor diferente no explorador de arquivos — isso indica que ela está sendo ignorada pelo Git.
10
Configurar o script de desenvolvimento no package.json
Definir o comando npm run dev para iniciar o servidor

Em vez de digitar nodemon index.js toda vez, vamos criar um atalho chamado dev. Scripts são atalhos de comandos definidos no package.json.

Abra o package.json e encontre a seção "scripts". Ela deve estar assim:

{
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  }
}

Substitua pela versão abaixo (mantendo a vírgula):

{
  "scripts": {
    "dev": "nodemon index.js",
    "start": "node index.js"
  }
}
Por que dois scripts? dev usa o nodemon (reinicia automático — para programar). start usa o node puro (para rodar em produção). Usaremos npm run dev durante todo o desenvolvimento.

4 Criando o primeiro servidor

Agora vem a parte mais importante do encontro: escrever o código do servidor. Vamos construir linha por linha, entendendo o que cada parte faz.

11
Criar o arquivo index.js
O ponto de entrada do servidor

Na barra lateral do VS Code, clique com o botão direito → New File → nomeie como index.js.

Esse arquivo será o ponto de entrada da sua aplicação — é o primeiro arquivo que o Node.js vai executar quando você iniciar o servidor.

✓ Estrutura do projeto até agora
📁 seu-projeto/ ├── 📁 node_modules/ ← ignorada pelo Git ├── 📄 index.js ← acabou de criar ← você está aqui ├── 📄 .gitignore ├── 📄 package.json ├── 📄 package-lock.json └── 📄 README.md
12
Escrever o código do servidor — linha por linha
Cada linha será explicada antes de você escrever

Abra o index.js e vamos escrever o código. Leia a explicação de cada bloco antes de digitá-lo.

Bloco 1 — Importar o Express e criar a aplicação

require('express') importa o Express que instalamos. É como dizer "quero usar essa ferramenta aqui". O resultado é guardado na constante express.
Na segunda linha, chamamos express() para criar nossa aplicação — o objeto app vai ser o centro de tudo: nele registramos rotas, middlewares e configurações.

const express = require('express')
const app = express()

Bloco 2 — Definir a porta

A porta é como um número de apartamento em um prédio: o prédio é o servidor, e cada porta é um "canal" por onde diferentes serviços se comunicam. A porta 3000 é a convenção para servidores em desenvolvimento. process.env.PORT permite que o servidor use uma porta diferente quando estiver em produção — veremos isso nos próximos encontros.

const PORT = process.env.PORT || 3000

Bloco 3 — Criar a primeira rota

Uma rota define o que acontece quando alguém faz uma requisição para um endereço específico. app.get('/', ...) significa: "quando alguém fizer uma requisição GET para o caminho / (a raiz), execute esta função".

A função recebe dois parâmetros:
— req (request) → tudo sobre a requisição que chegou (quem pediu, o que pediu, dados enviados)
— res (response) → o objeto que usamos para enviar a resposta de volta

res.json() envia a resposta no formato JSON — o formato padrão de comunicação entre sistemas web.

app.get('/', (req, res) => {
  res.json({ mensagem: 'Servidor funcionando! 🎉' })
})

Bloco 4 — Iniciar o servidor

app.listen(PORT, callback) diz ao servidor para começar a escutar requisições na porta definida. A função callback (o segundo argumento) é executada assim que o servidor estiver pronto — usamos ela para exibir uma mensagem de confirmação no terminal.

app.listen(PORT, () => {
  console.log(`Servidor rodando em http://localhost:${PORT}`)
})

O arquivo completo deve estar assim:

const express = require('express')
const app = express()

const PORT = process.env.PORT || 3000

app.get('/', (req, res) => {
  res.json({ mensagem: 'Servidor funcionando! 🎉' })
})

app.listen(PORT, () => {
  console.log(`Servidor rodando em http://localhost:${PORT}`)
})
13
Iniciar o servidor
O momento em que tudo ganha vida

No terminal, execute:

npm run dev
✓ Resultado esperado no terminal
[nodemon] starting `node index.js`
Servidor rodando em http://localhost:3000
O nodemon fica rodando continuamente (não volta para o prompt). Isso é normal — significa que o servidor está ativo e aguardando requisições.
Como parar o servidor: Pressione Ctrl + C no terminal. Para reiniciá-lo, execute npm run dev novamente. Com o nodemon, você raramente precisará fazer isso — ele reinicia sozinho quando você salva alterações.
Erro: "Cannot find module 'express'" O Express não foi instalado corretamente. Tente rodar npm install novamente e depois npm run dev.
Erro: "Error: listen EADDRINUSE :::3000" A porta 3000 já está sendo usada por outro servidor. Pare o servidor anterior com Ctrl+C ou reinicie o terminal.
14
Acessar o servidor no navegador
Ver a resposta chegando de verdade

O servidor está rodando, mas como acessá-lo? No Codespaces, quando um servidor é iniciado em uma porta, o VS Code detecta automaticamente e mostra uma notificação.

  1. Procure a notificação que aparece no canto inferior direito: "Your application on port 3000 is available"
  2. Clique em Open in Browser — ou acesse a aba Ports (junto ao terminal) e clique no ícone de globo ao lado da porta 3000
  3. O navegador abre com uma URL no formato https://xxxx-3000.app.github.dev
✓ Resultado esperado no navegador
{"mensagem":"Servidor funcionando! 🎉"}
Você está vendo a resposta JSON que seu servidor enviou. Isso é a sua API respondendo em tempo real!
Esse link funciona de qualquer lugar! A URL do Codespace é pública (enquanto o servidor estiver rodando). Você pode abrir no celular, compartilhar com alguém — é um servidor real na internet.
🚀 Aplique no seu Projeto

O servidor está funcionando com uma rota genérica. Agora é hora de personalizar para o seu projeto. Faça as três tarefas abaixo:

1 Personalize a rota principal
Altere a rota app.get('/') para retornar o nome do seu projeto e uma breve descrição. Exemplo: 
res.json({
  projeto: 'Sistema de Agendamentos',
  descricao: 'API para gerenciar horários e clientes',
  status: 'online'
})
2 Adicione uma segunda rota relacionada ao seu projeto
Pense em um recurso do seu projeto (produtos, agendamentos, tarefas, filmes…) e crie uma rota que retorna uma lista estática com 2-3 itens de exemplo. O banco de dados virá nos próximos encontros — por enquanto, os dados ficam direto no código. 
// Exemplo para um sistema de agendamentos:
app.get('/agendamentos', (req, res) => {
  res.json([
    { id: 1, cliente: 'Ana', horario: '09:00' },
    { id: 2, cliente: 'Carlos', horario: '10:30' }
  ])
})
3 Teste as duas rotas no navegador
Acesse a URL base e veja a resposta. Depois acrescente o caminho da segunda rota na URL (ex: .../agendamentos) e veja os dados retornados. Ambas devem responder com JSON correto.
⭐ Quer ir além? (opcional — só se terminar antes)

Estes desafios não são obrigatórios e não valem nota. São para quem terminou e quer explorar mais.

  • Adicione uma terceira rota para um segundo recurso do seu projeto (ex: além de /agendamentos, adicione /clientes)
  • Experimente retornar HTML: Troque res.json() por res.send('<h1>Olá!</h1>') em uma rota e veja a diferença no navegador. Por que usamos JSON em vez de HTML no backend?
  • Pesquise: O que é o código de status HTTP 200? E o 404? Por que eles importam? (Próximo encontro vamos trabalhar com isso)
