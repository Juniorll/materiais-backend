1 O problema — qualquer um acessa qualquer rota

O estado atual do seu projeto Qualquer pessoa com a URL da sua API pode fazer GET /tarefas, DELETE /pedidos/1 ou qualquer outra operação — sem se identificar. Em desenvolvimento isso é conveniente. Em produção, é um problema grave. Um sistema real precisa saber quem está fazendo cada requisição.

Antes de proteger qualquer rota, precisamos de dois conceitos:

Autenticação — quem é você? Verificar a identidade de quem está fazendo a requisição. "Você é mesmo o João? Prove." O processo é: o usuário apresenta uma credencial (email + senha), o sistema verifica, e emite uma prova de identidade (o token) que será usada nas próximas requisições.
Autorização — o que você pode fazer? Depois de saber quem é o usuário, verificar se ele tem permissão para aquela ação específica. "Você é o João — mas você pode deletar os pedidos da Maria?" Autenticação vem antes da autorização. Não há autorização sem autenticação.

2 O fluxo de autenticação com JWT

O fluxo tem quatro etapas. Acontece assim em qualquer sistema que usa JWT:

1
Cadastro
O usuário cria uma conta enviando nome, email e senha. O servidor criptografa a senha com bcrypt e salva no banco. A senha nunca é salva em texto puro.
↓
2
Login
O usuário envia email e senha. O servidor busca o usuário no banco, compara a senha com o hash usando bcrypt. Se conferir, gera e retorna um token JWT. Esse token é a "chave de entrada" para as próximas requisições.
↓
3
Enviar o token nas requisições
A partir do login, o cliente envia o token em todas as requisições que precisam de autenticação. O token vai no header HTTP: Authorization: Bearer <token>. O servidor não precisa consultar o banco a cada requisição — o token carrega a identidade do usuário.
↓
4
Verificar o token
O servidor verifica a assinatura do token. Se for válida (não foi adulterado, não expirou), extrai os dados do usuário (id, email) do payload e processa a requisição. Se inválido, retorna 401 Unauthorized.

3 A estrutura de um JWT

Um JWT é uma string com três partes separadas por ponto: header.payload.signature. Cada parte é codificada em Base64URL — não é criptografia, é apenas uma forma de representar JSON como texto.

Parte 1 — Header (vermelho no jwt.io)
Define o tipo do token e o algoritmo de assinatura. Quase sempre é o mesmo: tipo JWT, algoritmo HS256. 
{ "alg": "HS256", "typ": "JWT" }
Parte 2 — Payload (verde no jwt.io)
Os dados que você quer carregar no token — normalmente o id e email do usuário. Também tem campos reservados: iat (emitido em) e exp (expira em), em timestamp Unix. 
{
  "id": 1,
  "nome": "João Silva",
  "email": "joao@email.com",
  "iat": 1749571800,
  "exp": 1750176600
}

⚠️ O payload não é secreto. Qualquer pessoa pode decodificar o Base64URL e ler os dados. Nunca coloque senha, número de cartão ou qualquer dado sensível no payload.

Parte 3 — Signature (azul no jwt.io)
O que torna o JWT seguro. A assinatura é gerada com: HMAC-SHA256(header + "." + payload, JWT_SECRET). Se alguém modificar o payload (ex: trocar o id de 1 para 99), a assinatura não vai bater mais — o servidor rejeita o token com 401. A chave JWT_SECRET fica só no servidor e nunca é exposta.

4 Explorar no jwt.io

Acesse jwt.io e cole o token que o docente vai mostrar durante a demo. Explore:

→
No campo Encoded (esquerda): cole o token completo — as três partes coloridas ficam visíveis
→
No campo Decoded (direita): veja o header (vermelho), o payload (verde) e a assinatura (azul)
→
Tente modificar o payload diretamente — troque o id ou o email. Observe que o token codificado muda mas a assinatura fica marcada como inválida
O jwt.io é uma ferramenta de inspeção, não de geração de tokens reais Você pode usar para debugar e entender a estrutura — mas tokens usados em produção são gerados pela aplicação com a JWT_SECRET que fica no servidor. O jwt.io não sabe a sua secret, então não consegue verificar a assinatura de tokens reais (a menos que você cole a secret lá — o que nunca deve fazer com tokens de produção).

# .env — adicionar esta linha
JWT_SECRET=sua_chave_secreta_longa_e_aleatoria_aqui

Use uma string longa e aleatória (pelo menos 32 caracteres). Não use "123456" ou "secret". Em produção, use uma chave gerada com crypto.randomBytes(32).toString('hex').

1 Adulterar um token e testar a rejeição

No jwt.io, pegue um token válido e edite o payload — troque o id por um número diferente, ou adicione um campo novo. Copie o token gerado (com assinatura inválida) e teste no Postman na rota GET /auth/eu do sistema demonstrado pelo docente. O servidor deve retornar 401. Entender por que isso acontece é entender por que JWT é seguro.

2 Entender os claims reservados do JWT

O payload pode conter qualquer campo, mas o padrão JWT define alguns claims reservados com significado especial:

iat (issued at) — timestamp de quando o token foi emitido
exp (expiration) — timestamp de quando o token expira. Tokens expirados são rejeitados automaticamente pelo jwt.verify()
sub (subject) — identifica o sujeito do token, geralmente o id do usuário
iss (issuer) — identifica quem emitiu o token (nome da aplicação)

Explore esses campos no jwt.io usando tokens com tempo de expiração curto (ex: expiresIn: '10s') e observe o comportamento quando o token expira.

Estes tópicos não são cobrados e não entram no projeto.
Sessions vs JWT — stateful vs stateless
A autenticação tradicional usa sessions no servidor: ao fazer login, o servidor cria uma sessão, salva no banco/memória e devolve um session ID (cookie). A cada requisição, o servidor consulta o banco para verificar se a sessão é válida. JWT é stateless: o servidor não precisa armazenar nada — toda a informação está no token. Cada abordagem tem tradeoffs: sessions são mais fáceis de invalidar (logout imediato), JWT escala melhor horizontalmente (sem compartilhamento de estado entre servidores).
Refresh Tokens — manter a sessão sem expor o access token
Access tokens JWT têm vida curta (minutos ou horas) por segurança — se vazarem, o dano é limitado. Para manter o usuário logado sem pedir senha toda hora, usa-se um refresh token de vida longa (dias/semanas), salvo no servidor. Quando o access token expira, o cliente troca o refresh token por um novo access token sem reautenticar. É o padrão em sistemas como Google, GitHub e Spotify.
OAuth 2.0 — "entrar com Google / GitHub"
OAuth 2.0 é um protocolo de autorização que permite que um usuário conceda acesso a uma aplicação sem compartilhar sua senha. É o que acontece quando você clica em "Entrar com Google" — o Google autentica o usuário e entrega um token para a sua aplicação. No Node.js, a biblioteca passport.js implementa dezenas de provedores OAuth.
