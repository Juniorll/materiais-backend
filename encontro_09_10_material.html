1 O problema e a solução PARTE 1

⚠ O risco de segredos no código
Imagine que você adiciona a senha do banco de dados diretamente no código: password: 'minha_senha_123'. O código vai para o GitHub. O repositório é público. Agora qualquer pessoa no mundo pode ver essa senha. Isso acontece com frequência — senhas e chaves de API expostas em repositórios são uma das vulnerabilidades mais comuns em projetos de desenvolvedores iniciantes.
A solução: variáveis de ambiente Uma variável de ambiente é um valor que existe no ambiente onde o código roda — no sistema operacional, não no código em si. O arquivo .env é um arquivo local que define essas variáveis para o ambiente de desenvolvimento. O pacote dotenv carrega esse arquivo e coloca as variáveis em process.env. O código lê de process.env. O arquivo .env fica no .gitignore — nunca vai para o GitHub.
Desenvolvimento vs Produção No desenvolvimento (seu Codespace), as variáveis vêm do arquivo .env carregado pelo dotenv. Em produção (Railway), as variáveis são configuradas diretamente no painel da plataforma — sem arquivo .env. O código não muda — ele sempre lê de process.env. O que muda é de onde esses valores vêm.

Na raiz do projeto, crie um arquivo chamado .env (com o ponto na frente, sem extensão). Por convenção, nomes de variáveis de ambiente são escritos em maiúsculas com underscore.

Uma linha no topo do arquivo — deve ser a primeira coisa executada

Adicione o require('dotenv').config() como primeira linha do arquivo — antes de qualquer outro código que use process.env.

1 Rota de Health Check (verificação de saúde)

Em produção, ferramentas de monitoramento "perguntam" periodicamente para o servidor se ele está vivo e respondendo — isso se chama health check. É uma rota simples que qualquer API profissional tem.

Como fazer:

  1. Crie uma rota GET /saude diretamente no index.js (não é um recurso do domínio, então não precisa estar em routes/)
  2. Retorne um JSON com três informações: o status ('ok'), o tempo que o servidor está rodando, e o horário atual
  3. Para o tempo de execução, use process.uptime() — uma função nativa do Node.js que retorna os segundos desde que o processo começou
app.get('/saude', (req, res) => {
  res.json({
    status: 'ok',
    uptime: process.uptime(), // segundos rodando
    timestamp: new Date().toISOString()
  })
})

Teste no Postman e chame a rota duas vezes com alguns segundos de intervalo — o uptime deve aumentar entre as chamadas, confirmando que está calculando corretamente.

2 Logs condicionais com NODE_ENV

O middleware de log que você criou no E06 é ótimo para desenvolvimento, mas em produção ele poluiria os registros do servidor com informação que ninguém vai ler. A prática comum é usar uma variável de ambiente chamada NODE_ENV para diferenciar os dois contextos — e hoje você já sabe exatamente como adicionar isso ao seu .env.

Como fazer:

  1. No .env, adicione a linha NODE_ENV=development
  2. No middleware de log (E06), envolva o console.log numa condição: só registra se process.env.NODE_ENV for diferente de 'production'
  3. Atenção: o next() deve continuar sendo chamado sempre — com ou sem log, a requisição precisa seguir adiante
  4. Para testar: troque o valor no .env para production, salve, e veja o log parar de aparecer. Depois volte para development
app.use((req, res, next) => {
  if (process.env.NODE_ENV !== 'production') {
    const horario = new Date().toLocaleTimeString('pt-BR')
    console.log(`[${horario}] ${req.method} ${req.path}`)
  }
  next() // chamado sempre, com ou sem log
})

Testes automatizados com Jest e Supertest
Até agora, toda vez que você altera o código, você testa manualmente no Postman — clicando em cada requisição, uma por uma. Em projetos profissionais, isso é automatizado: Jest é um framework de testes para JavaScript, e Supertest permite simular requisições HTTP para testar rotas Express sem precisar do servidor rodando numa porta real. Você escreve o teste uma vez ("a rota GET /filmes deve retornar status 200 e uma lista"), e pode rodá-lo quantas vezes quiser com um único comando.
→ jestjs.io e github.com/ladjs/supertest
CI/CD com GitHub Actions
CI significa Continuous Integration — Integração Contínua. A ideia: toda vez que você faz git push, um robô do GitHub roda automaticamente os testes do seu projeto e te avisa se algo quebrou — antes mesmo de você notar. O GitHub Actions é o recurso que faz isso, configurado por um arquivo de receita dentro da pasta .github/workflows/ do repositório. É amplamente usado em equipes de desenvolvimento para garantir que ninguém envie código quebrado sem perceber.
→ docs.github.com/pt/actions
