1 Por que organizar e o que é o Express Router

O problema que vamos resolver Um projeto com um único recurso já tem ~60 linhas de rotas. Com dois recursos são ~120. Com autenticação, validações e mais recursos, o index.js chega facilmente a 300-400 linhas — um arquivo difícil de navegar, difícil de manter e difícil de trabalhar em equipe. A solução padrão é separar as rotas de cada recurso em seu próprio arquivo.
O que é o Express Router O Express Router é como uma "mini-aplicação" dentro da sua aplicação. Com ele, você cria um objeto router que funciona exatamente como o app — registra rotas com router.get(), router.post(), etc. A diferença é que o router é exportado do arquivo e registrado no index.js com um prefixo de caminho. Tudo que estava em app.get('/produtos', ...) vira router.get('/', ...) — o prefixo /produtos fica no index.js.
Antes — tudo no index.js
app.get('/produtos', ...)
app.post('/produtos', ...)
app.get('/produtos/:id', ...)
app.put('/produtos/:id', ...)
app.delete('/produtos/:id', ...)
Depois — routes/produtos.js
router.get('/', ...)
router.post('/', ...)
router.get('/:id', ...)
router.put('/:id', ...)
router.delete('/:id', ...)
Quando você registra o router com app.use('/produtos', router), o Express automaticamente adiciona o prefixo /produtos a todas as rotas do router. Por isso dentro do arquivo de rotas usamos apenas / e /:id — sem repetir o nome do recurso.

2 Manual — refatorando o projeto passo a passo

Regra de ouro da refatoração Não delete as rotas antigas do index.js até o novo arquivo estar funcionando e testado. Trabalhe em paralelo: crie o novo arquivo, teste, confirme que funciona — só então remova as rotas do index.js.

2
Configurar o Express Router no novo arquivo
Criar o router e mover os dados para o novo arquivo

Abra o arquivo recém-criado em routes/. As primeiras linhas definem o router — o mecanismo que vai substituir o app para registrar as rotas:

const express = require('express')
const router = express.Router()

// Dados temporários (o array do seu recurso)
// Cole aqui o array que estava no index.js
const itens = [
  // seus dados de exemplo
]

// As rotas virão aqui nos próximos passos

module.exports = router

A última linha — module.exports = router — exporta o router para que o index.js possa importá-lo. Deve ser sempre a última linha do arquivo.

Por que mover o array para cá? Os dados pertencem ao contexto das rotas daquele recurso. Quando as rotas estiverem aqui, elas precisam acessar o array — então o array fica no mesmo arquivo. Quando migrarmos para banco de dados no Bloco 3, o array será substituído pelas queries SQL, também aqui.
3
Mover as rotas do index.js para o arquivo de rotas
Substituir app. por router. e remover o prefixo dos caminhos

Copie cada rota do index.js para o novo arquivo, fazendo duas mudanças em cada uma:

  • Substitua app. por router.
  • Remova o prefixo do recurso do caminho: '/produtos' vira '/' e '/produtos/:id' vira '/:id'

Exemplo completo do arquivo de rotas após mover tudo:

const express = require('express')
const router = express.Router()

const produtos = [
  { id: 1, nome: 'Produto A', preco: 29.90 },
  { id: 2, nome: 'Produto B', preco: 49.90 }
]

// GET / → lista todos  (era GET /produtos no index.js)
router.get('/', (req, res) => {
  res.json(produtos)
})

// GET /:id → busca por ID  (era GET /produtos/:id)
router.get('/:id', (req, res) => {
  const id = Number(req.params.id)
  const produto = produtos.find(p => p.id === id)
  if (!produto) return res.status(404).json({ erro: 'Não encontrado' })
  res.json(produto)
})

// POST / → cria novo  (era POST /produtos)
router.post('/', (req, res) => {
  const { nome, preco } = req.body
  const novo = { id: produtos.length + 1, nome, preco }
  produtos.push(novo)
  res.status(201).json(novo)
})

// PUT /:id → atualiza  (era PUT /produtos/:id)
router.put('/:id', (req, res) => {
  const id = Number(req.params.id)
  const index = produtos.findIndex(p => p.id === id)
  if (index === -1) return res.status(404).json({ erro: 'Não encontrado' })
  const { nome, preco } = req.body
  produtos[index] = { id, nome, preco }
  res.json(produtos[index])
})

// DELETE /:id → remove  (era DELETE /produtos/:id)
router.delete('/:id', (req, res) => {
  const id = Number(req.params.id)
  const index = produtos.findIndex(p => p.id === id)
  if (index === -1) return res.status(404).json({ erro: 'Não encontrado' })
  produtos.splice(index, 1)
  res.status(204).send()
})

module.exports = router
4
Importar e registrar o router no index.js
Conectar o arquivo de rotas ao servidor principal

Abra o index.js e adicione o import e o registro do router. Coloque após os middlewares e antes do app.listen:

// Importar o arquivo de rotas
const produtosRoutes = require('./routes/produtos')

// Registrar com o prefixo do recurso
app.use('/produtos', produtosRoutes)

O ./routes/produtos é o caminho relativo ao index.js. O ./ significa "na mesma pasta que este arquivo" — então o Node vai procurar em routes/produtos.js.

A linha app.use() com o router funciona igual ao middleware O app.use('/produtos', produtosRoutes) é tecnicamente um middleware também — ele intercepta toda requisição que começa com /produtos e delega para o router. Tudo é middleware no Express.
5
Testar no Postman — tudo deve funcionar igual
A refatoração é transparente: o comportamento não muda

Com as rotas antigas ainda no index.js (ainda não deletamos), teste o CRUD completo no Postman. As URLs são exatamente as mesmas de antes:

  • GET /produtos → lista todos
  • POST /produtos → cria novo
  • GET /produtos/1 → busca por ID
  • PUT /produtos/1 → atualiza
  • DELETE /produtos/1 → remove
✓ Resultado esperado
Todas as rotas respondem corretamente — exatamente como antes da refatoração. Se algo não funcionar, resolva antes de avançar para o próximo passo.
Cannot GET /produtos — 404 em todas as rotas O router não foi registrado no index.js ou o require está com o caminho errado. Confirme que a linha app.use('/produtos', produtosRoutes) está no index.js e que o caminho ./routes/produtos corresponde ao arquivo criado.
Erro "Cannot find module './routes/produtos'" O nome do arquivo ou o caminho está diferente. Verifique: o arquivo está dentro de routes/? O nome no require bate com o nome do arquivo?
6
Limpar o index.js — remover as rotas antigas
Só após confirmar que o novo arquivo funciona

Agora que o novo arquivo de rotas está testado e funcionando, remova as rotas do index.js. Remova também o array de dados que foi movido para o arquivo de rotas.

O index.js final deve ficar assim — limpo e focado em configuração:

const express = require('express')
const app = express()
const PORT = process.env.PORT || 3000

// Middlewares
app.use(express.json())
app.use((req, res, next) => {
  const horario = new Date().toLocaleTimeString('pt-BR')
  console.log(`[${horario}] ${req.method} ${req.path}`)
  next()
})

// Rotas
const produtosRoutes = require('./routes/produtos')
app.use('/produtos', produtosRoutes)

// Rota raiz (opcional — informações do projeto)
app.get('/', (req, res) => {
  res.json({ projeto: 'Meu Projeto', status: 'online' })
})

app.listen(PORT, () => {
  console.log(`Servidor rodando em http://localhost:${PORT}`)
})

Salve e teste no Postman mais uma vez — tudo deve continuar funcionando.

Controllers — separando a lógica das rotas
Hoje separamos as rotas do index.js. O próximo passo natural é separar a lógica das rotas também. Um arquivo controllers/produtos.js conteria as funções (listar, buscar, criar, atualizar, deletar), e o arquivo de rotas apenas registraria quais funções respondem a quais caminhos. Isso é o padrão MVC (Model-View-Controller) simplificado — amplamente usado no mercado e o que você vai encontrar na maioria dos projetos Node.js profissionais.
→ MDN — O que é MVC
NestJS — organização por convenção
O Express é um framework minimalista: ele dá ferramentas, mas não impõe estrutura. O NestJS é um framework construído em cima do Express (ou Fastify) que impõe uma estrutura de pastas bem definida — modules, controllers, services, decorators. É o framework mais usado em projetos Node.js empresariais que precisam de uma base sólida desde o início. Se você continuar em backend depois desta UC, vai encontrar NestJS com frequência.
