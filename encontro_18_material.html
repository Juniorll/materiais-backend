1 Por que validar — e o que é o status 400

O problema sem validação Sem validação, o banco aceita qualquer coisa: preço vazio, nome com só espaços, quantidade negativa. Quando algo quebra, o erro vem do banco — uma mensagem técnica que o cliente da API não entende. Com validação, a rota rejeita antes de tentar salvar e responde com 400 Bad Request e uma mensagem que explica exatamente o que está errado.
Onde validar — e onde não validar A validação fica nas rotas, antes do INSERT ou UPDATE. Não valide dentro do database.js — lá é o banco, não a lógica de negócio. O banco já tem suas próprias restrições (NOT NULL, CHECK) como segunda linha de defesa. A validação nas rotas é a primeira.

2 Padrões de validação

1
Os três padrões mais usados
Aprenda os padrões antes de aplicar no projeto
Padrão 1 — Campo obrigatório (texto)
// Rejeita: undefined, null, string vazia, string só com espaços
if (!nome || nome.trim() === '') {
  return res.status(400).json({ erro: 'O campo "nome" é obrigatório' })
}
Padrão 2 — Número com range válido
// Rejeita: não-número, negativo, acima do máximo
if (preco === undefined || typeof preco !== 'number' || preco < 0) {
  return res.status(400).json({ erro: 'preco deve ser um número maior ou igual a zero' })
}

// Nota de 1 a 5
if (!nota || !Number.isInteger(nota) || nota < 1 || nota > 5) {
  return res.status(400).json({ erro: 'nota deve ser um inteiro entre 1 e 5' })
}
Padrão 3 — Valor de lista permitida
// Rejeita qualquer valor fora da lista
const statusPermitidos = ['pendente', 'confirmado', 'cancelado']
if (status && !statusPermitidos.includes(status)) {
  return res.status(400).json({
    erro: `status deve ser um de: ${statusPermitidos.join(', ')}`
  })
}
Retornar múltiplos erros de uma vez Em vez de retornar ao primeiro erro encontrado, colete todos e retorne juntos — o cliente da API fica mais informado: 
const erros = []
if (!nome?.trim()) erros.push('nome é obrigatório')
if (preco === undefined) erros.push('preco é obrigatório')
if (erros.length > 0) return res.status(400).json({ erros })

3 Helper de validação reutilizável

2
Criar helpers/validacao.js
Funções que as rotas vão importar — evita repetir a mesma lógica em cada arquivo

Crie a pasta helpers/ na raiz do projeto e o arquivo validacao.js dentro dela:

// helpers/validacao.js

// Verifica se todos os campos obrigatórios estão presentes e não vazios
function validarObrigatorios(dados, campos) {
  return campos
    .filter(campo => {
      const valor = dados[campo]
      return valor === undefined || valor === null ||
             (typeof valor === 'string' && valor.trim() === '')
    })
    .map(campo => `O campo "${campo}" é obrigatório`)
}

// Verifica se um número está dentro de um intervalo
function validarRange(campo, valor, min, max) {
  if (typeof valor !== 'number' || valor < min || valor > max) {
    return `${campo} deve ser um número entre ${min} e ${max}`
  }
  return null
}

// Verifica se um valor está em uma lista de permitidos
function validarLista(campo, valor, permitidos) {
  if (valor !== undefined && !permitidos.includes(valor)) {
    return `${campo} deve ser um de: ${permitidos.join(', ')}`
  }
  return null
}

module.exports = { validarObrigatorios, validarRange, validarLista }

4 Aplicar validação nas rotas do projeto

Localize o seu projeto e adicione a validação nas rotas POST e PUT da tabela principal. O padrão é sempre: importar o helper, montar o array de erros, retornar 400 se houver erros, prosseguir se estiver tudo ok.

3
Validação nas rotas — código por projeto
Adicione ao topo dos arquivos de rota e adapte as regras conforme as decisões de modelagem
🛒 Sistema de Pedidos / Loja — routes/produtos.js
const { validarObrigatorios, validarRange } = require('../helpers/validacao')

// POST /produtos
router.post('/', (req, res, next) => {
  try {
    const { nome, preco, estoque } = req.body

    const erros = validarObrigatorios(req.body, ['nome', 'preco'])
    const erroPreco = validarRange('preco', preco, 0, Infinity)
    if (erroPreco) erros.push(erroPreco)
    if (erros.length > 0) return res.status(400).json({ erros })

    const resultado = db.prepare('INSERT INTO produtos (nome, preco, estoque) VALUES (?, ?, ?)')
      .run(nome.trim(), preco, estoque ?? 0)
    res.status(201).json(db.prepare('SELECT * FROM produtos WHERE id = ?').get(resultado.lastInsertRowid))
  } catch (err) { next(err) }
})
📅 Agenda / Agendamentos — routes/clientes.js

    res.status(201).json(db.prepare('SELECT * FROM clientes WHERE id = ?').get(resultado.lastInsertRowid))
  } catch (err) { next(err) }
})

4
Testar os cenários de erro no Postman

Para cada rota validada, teste estes cenários e confirme que a resposta é 400 com a mensagem correta:

1. Enviar o body sem o campo obrigatório → { "erros": ["O campo \"nome\" é obrigatório"] }

2. Enviar o campo obrigatório com string vazia "" → deve rejeitar da mesma forma

3. Enviar número fora do range (preco negativo, nota = 6) → { "erros": ["preco deve ser..."] }

4. Enviar dados completamente válidos → deve retornar 201 normalmente

⭐ Quer ir além? (opcional)
1 Validar formato de email com regex

O email.includes('@') rejeita os casos mais óbvios mas não é suficiente. Uma regex básica cobre a maioria dos casos reais:

function emailValido(email) {
  const regex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/
  return regex.test(email)
}

// Uso:
if (email && !emailValido(email)) {
  erros.push('email com formato inválido')
}

Adicione essa função ao helpers/validacao.js e exporte junto com as outras.

2 Middleware de validação — reutilizar entre rotas

Se várias rotas compartilham as mesmas regras de validação, você pode criar um middleware de validação e registrá-lo antes da rota:

// Middleware que valida antes de passar para a rota
function validarProduto(req, res, next) {
  const erros = validarObrigatorios(req.body, ['nome', 'preco'])
  if (erros.length > 0) return res.status(400).json({ erros })
  next() // Passa para o próximo handler
}

// Registrar antes das rotas POST e PUT
router.post('/', validarProduto, (req, res, next) => { /* ... */ })
router.put('/:id', validarProduto, (req, res, next) => { /* ... */ })

Zod — validação com schema declarativo
Em vez de escrever validação manualmente para cada campo, o Zod permite declarar um schema: z.object({ nome: z.string().min(1), preco: z.number().min(0) }) — e o próprio Zod valida o objeto e gera as mensagens de erro. É o padrão em muitos projetos Node.js modernos com TypeScript.
→ zod.dev — documentação
express-validator — validação integrada ao Express
Uma biblioteca de validação especificamente desenhada para o Express. Permite declarar regras como middleware diretamente na definição da rota e extrair os erros de forma padronizada. Alternativa popular ao Zod para quem trabalha só com JavaScript.
→ express-validator.github.io
